메모리 할당

Pintos에는 두 가지 메모리 allocator가 있습니다. 하나는 page 단위로 메모리를 할당하고, 다른 하나는 임의 크기의 block을 할당할 수 있습니다.

Page Allocator

include/threads/palloc.h에 선언된 page allocator는 page 단위로 메모리를 할당합니다. 보통 한 번에 한 page씩 메모리를 할당하는 데 가장 자주 사용되지만, 여러 개의 연속된 page를 한 번에 할당할 수도 있습니다.

page allocator는 자신이 할당하는 메모리를 kernel pool과 user pool이라는 두 pool로 나눕니다. 기본적으로 각 pool은 1 MB 위쪽의 시스템 메모리 절반씩을 받지만, ul 커널 command-line option으로 이 분할을 바꿀 수 있습니다. 할당 요청은 둘 중 한 pool에서 가져옵니다. 한 pool이 비어도 다른 pool에는 free page가 남아 있을 수 있습니다. user pool은 사용자 프로세스용 메모리 할당에 사용해야 하고, kernel pool은 그 밖의 모든 할당에 사용해야 합니다. 이 구분은 프로젝트 3부터 중요해집니다. 그 전까지는 모든 할당을 kernel pool에서 하면 됩니다.

각 pool의 사용 상태는 bitmap으로 추적됩니다. pool 안의 page 하나당 bit 하나를 사용합니다. n개의 page를 할당하라는 요청이 들어오면 bitmap에서 false로 설정된 연속 n개 bit를 찾습니다. 이 bit들은 해당 page가 비어 있음을 뜻합니다. 그런 다음 그 bit들을 true로 설정해 사용 중이라고 표시합니다. 이는 "first fit" 할당 전략입니다.

page allocator는 단편화의 영향을 받습니다. 즉 free page가 n개 이상 있더라도, 그 free page들이 사용 중인 page들 사이에 흩어져 있으면 연속된 n개 page를 할당하지 못할 수 있습니다. 심한 경우 pool page의 절반이 비어 있어도 연속된 2개 page를 할당하지 못할 수 있습니다. 단일 page 요청은 단편화 때문에 실패할 수 없으므로, 여러 연속 page를 요청하는 일은 가능한 한 제한해야 합니다.

page는 interrupt context에서 할당할 수 없지만, 해제할 수는 있습니다.

page가 해제되면 디버깅을 돕기 위해 그 모든 바이트가 0xcc로 채워집니다. 디버깅 도구를 참고하세요.

page allocator의 타입과 함수는 아래와 같습니다.

void *palloc_get_page (enum palloc_flags flags)
void *palloc_get_multiple (enum palloc_flags flags, size_t page_cnt)

각각 page 하나 또는 page_cnt개의 연속 page를 얻어 반환합니다. page를 할당할 수 없으면 null pointer를 반환합니다.

flags 인자는 다음 flag들의 조합일 수 있습니다.

  • PAL_ASSERT

    page를 할당할 수 없으면 커널을 panic시킵니다. 이는 커널 초기화 중에만 적절합니다. 사용자 프로세스가 커널을 panic시킬 수 있게 해서는 안 됩니다.

  • PAL_ZERO

    할당된 page를 반환하기 전에 모든 바이트를 0으로 만듭니다. 이 flag가 설정되어 있지 않으면 새로 할당된 page의 내용은 예측할 수 없습니다.

  • PAL_USER

    user pool에서 page를 얻습니다. 설정되어 있지 않으면 page는 kernel pool에서 할당됩니다.

void palloc_free_page (void *page)
void palloc_free_multiple (void *pages, size_t page_cnt)

pages에서 시작하는 page 하나 또는 page_cnt개의 연속 page를 각각 해제합니다. 모든 page는 palloc_get_page() 또는 palloc_get_multiple()로 얻은 것이어야 합니다.

Block Allocator

threads/malloc.h에 선언된 block allocator는 임의 크기의 block을 할당할 수 있습니다. 이 allocator는 앞 절에서 설명한 page allocator 위에 계층화되어 있습니다. block allocator가 반환하는 block은 kernel pool에서 얻습니다.

block allocator는 메모리 할당에 두 가지 전략을 사용합니다. 첫 번째 전략은 1 kB 이하의 block에 적용됩니다. 이는 page size의 1/4입니다. 이런 할당은 가장 가까운 2의 거듭제곱 크기나 16바이트 중 더 큰 값으로 올림됩니다. 그런 다음 해당 크기의 할당에만 쓰이는 page로 묶입니다.

두 번째 전략은 1 kB보다 큰 block에 적용됩니다. 이런 할당은 약간의 overhead를 더한 뒤 가장 가까운 page 크기로 올림됩니다. 그런 다음 block allocator가 page allocator에 그 수만큼의 연속 page를 요청합니다.

두 경우 모두 요청한 할당 크기와 실제 block 크기의 차이는 낭비됩니다. 실제 운영체제라면 이런 낭비를 최소화하도록 allocator를 세심하게 조정하겠지만, Pintos 같은 교육용 시스템에서는 중요하지 않습니다.

page allocator에서 page를 얻을 수 있는 한 작은 할당은 항상 성공합니다. 대부분의 작은 할당은 page allocator에서 새 page를 전혀 요구하지 않습니다. 이미 할당된 page의 일부로 충족되기 때문입니다. 하지만 큰 할당은 항상 page allocator를 호출해야 하며, 연속 page가 두 개 이상 필요한 할당은 앞 절에서 설명한 것처럼 단편화 때문에 실패할 수 있습니다. 따라서 코드에서 큰 할당, 특히 각각 약 4 kB를 넘는 할당의 수를 최소화해야 합니다.

block이 해제되면 디버깅을 돕기 위해 그 모든 바이트가 0xcc로 채워집니다. 디버깅 도구를 참고하세요.

block allocator는 interrupt context에서 호출할 수 없습니다.

block allocator 함수는 아래와 같습니다. 인터페이스는 같은 이름의 표준 C 라이브러리 함수와 같습니다.

void *malloc (size_t size)

kernel pool에서 최소 size 바이트 길이의 새 block을 얻어 반환합니다. size가 0이거나 메모리를 사용할 수 없으면 null pointer를 반환합니다.

void *calloc (size_t a, size_t b)

kernel pool에서 최소 a * b 바이트 길이의 새 block을 얻어 반환합니다. block의 내용은 0으로 지워집니다. a나 b가 0이거나 메모리가 부족하면 null pointer를 반환합니다.

void *realloc (void *block, size_t new_size)

block을 new_size 바이트로 resize하려고 시도합니다. 이 과정에서 block이 이동할 수도 있습니다. 성공하면 새 block을 반환하며, 이 경우 이전 block에는 더 이상 접근하면 안 됩니다. 실패하면 null pointer를 반환하고 이전 block은 계속 유효합니다. block이 null인 호출은 malloc()과 같고, new_size가 0인 호출은 free()와 같습니다.

void free (void *block)

malloc(), calloc(), 또는 realloc()이 이전에 반환했고 아직 해제되지 않은 block을 해제합니다.
