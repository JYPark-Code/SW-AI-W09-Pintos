원문 보기
KAIST Pintos GitBook 원문을 한국어로 옮긴 로컬 읽기본입니다. 코드, 명령어, 함수명, 파일명은 의도적으로 원문 형태를 최대한 유지했습니다. 의미가 애매한 부분은 원문 링크와 함께 확인하세요.

디버깅 도구

Pintos를 디버깅할 때 사용할 수 있는 도구는 많습니다. 이 부록에서는 그중 몇 가지를 소개합니다.

printf()

printf()의 가치를 과소평가하지 마세요. Pintos의 printf() 구현은 커널 안 거의 어디서든 호출할 수 있도록 되어 있습니다. kernel thread 안이든 interrupt handler 안이든, 어떤 lock을 잡고 있는지와 거의 무관하게 사용할 수 있습니다.

printf()는 단순히 데이터를 찍어 보는 것 이상으로 유용합니다. kernel이 유용한 오류 메시지 없이 crash하거나 panic하더라도, 문제가 언제 어디서 발생하는지 좁혀 가는 데 도움이 됩니다. 의심되는 코드 구간 여러 곳에 서로 다른 문자열을 출력하는 printf() 호출을 흩뿌려 두는 방식입니다. 예를 들어 <1>, <2> 같은 문자열을 출력하게 합니다. <1>조차 보이지 않으면 그 지점 이전에 문제가 생긴 것이고, <1>은 보이지만 <2>가 보이지 않으면 두 지점 사이에서 문제가 생긴 것입니다. 이렇게 알게 된 내용을 바탕으로 더 좁은 의심 구간에 다시 printf()를 넣으면, 결국 문제를 하나의 문장까지 좁힐 수 있습니다. 관련 기법은 Triple Faults 절을 참고하세요.

ASSERT

Assertion은 문제가 더 크게 드러나기 전에 초기에 잡아낼 수 있어서 유용합니다. 이상적으로는 각 함수가 시작될 때 인자의 유효성을 검사하는 assertion들을 두는 것이 좋습니다. 단, 함수의 local variable initializer는 assertion 검사보다 먼저 평가되므로, initializer 안에서 인자가 유효하다고 가정하지 않도록 주의해야 합니다. 또한 문제가 생길 가능성이 높은 함수 본문 곳곳에도 assertion을 넣을 수 있습니다. 특히 loop invariant를 검사할 때 유용합니다.

Pintos는 assertion 검사용으로 <debug.h>에 정의된 ASSERT macro를 제공합니다.

#define ASSERT(expression) { /* Omit details */ }

expression의 값을 검사합니다. 값이 0, 즉 false로 평가되면 kernel panic이 발생합니다. Panic message에는 실패한 expression, 파일과 줄 번호, 그리고 문제 위치를 찾는 데 도움이 되는 backtrace가 포함됩니다. 자세한 내용은 Backtraces를 참고하세요.

Function과 Parameter Attribute

<debug.h>에 정의된 다음 macro들은 함수나 함수 parameter의 특수한 속성을 compiler에게 알려 줍니다. 확장 결과는 GCC 전용입니다.

#define UNUSED { /* Omit details */ }

함수 parameter 뒤에 붙여, 해당 parameter가 함수 안에서 사용되지 않을 수도 있음을 compiler에게 알려 줍니다. 그렇지 않으면 발생할 warning을 억제합니다.

#define NO_RETURN { /* Omit details */ }

함수 prototype 뒤에 붙여, 해당 함수가 절대 반환하지 않는다는 것을 compiler에게 알려 줍니다. 이를 통해 compiler가 warning과 code generation을 더 정확히 조정할 수 있습니다.

#define NO_INLINE { /* Omit details */ }

함수 prototype 뒤에 붙여, compiler가 해당 함수를 inline으로 내보내지 않도록 합니다. 때때로 backtrace 품질을 높이는 데 유용합니다.

#define PRINTF_FORMAT(format, first) { /* Omit details */ }

함수 prototype 뒤에 붙여, 해당 함수가 printf()와 비슷한 format string을 받는다는 사실을 compiler에게 알려 줍니다. format은 1부터 세었을 때 format string 인자의 번호이고, first는 대응되는 값 인자들이 시작되는 번호입니다. 덕분에 잘못된 타입의 인자를 넘겼을 때 compiler가 알려 줄 수 있습니다.

Backtraces

Kernel이 panic하면 "backtrace"를 출력합니다. Backtrace는 panic 시점에 실행 중이던 함수들 안의 주소 목록으로, 프로그램이 현재 위치에 어떻게 도달했는지 요약한 것입니다. 코드의 임의 위치에서 backtrace를 출력하고 싶다면 <debug.h>에 prototype이 있는 debug_backtrace()를 호출할 수 있습니다. 역시 <debug.h>에 선언된 debug_backtrace_all()은 모든 스레드의 backtrace를 출력합니다.

Backtrace 안의 주소는 해석하기 어려운 raw hexadecimal number로 나옵니다. Pintos는 이 주소들을 함수 이름과 source file line number로 바꿔 주는 backtrace 도구를 제공합니다. 첫 번째 인자로 kernel.o 이름을 넘기고, 나머지 인자로 backtrace를 구성하는 hexadecimal number들을 0x prefix까지 포함해 넘기면 됩니다. 출력은 각 주소에 대응되는 함수 이름과 source file line number입니다.

변환된 backtrace가 깨져 보이거나 말이 되지 않는다면, 예를 들어 function A가 function B 위에 나오는데 B가 A를 호출하지 않는다면, kernel thread의 stack을 손상시키고 있다는 강한 신호입니다. backtrace가 stack에서 추출되기 때문입니다. 또는 backtrace에 넘긴 kernel.o가 실제 backtrace를 만든 kernel과 다른 것일 수도 있습니다.

손상이 없어도 backtrace가 헷갈릴 수 있습니다. Compiler optimization이 예상 밖의 결과를 만들 수 있기 때문입니다. 어떤 함수가 마지막 동작으로 다른 함수를 호출하는 tail call을 하면, 호출한 함수가 backtrace에 아예 나타나지 않을 수 있습니다. 비슷하게 function A가 반환하지 않는 function B를 호출하면, compiler가 최적화한 결과 A 대신 관련 없는 function C가 backtrace에 나타날 수 있습니다. C는 단지 메모리에서 A 바로 뒤에 놓인 함수일 뿐입니다. threads project의 test failure backtrace에서 이런 현상을 자주 볼 수 있습니다.

예제

다음은 예제입니다. Pintos가 file system project의 실제 제출물에서 다음 call stack을 출력했다고 합시다.

Call stack: 0x800426eff 0x8004202fb 0x80042dc22 0x80042cf67 0x800422319
0x80042325a 0x40012c 0x400a96 0x400ac8.

그러면 아래처럼 backtrace 정보를 command line에 붙여 넣어 backtrace utility를 실행합니다. 여기서는 kernel.o가 현재 디렉터리에 있다고 가정합니다. 아래 내용은 문서 폭 때문에 두 줄로 보이지만, 실제 shell에서는 한 줄 command로 입력해야 합니다.

backtrace 0x800426eff 0x8004202fb 0x80042dc22 0x80042cf67 0x800422319
0x80042325a 0x400012c 0x400a96 0x4000ac8

그러면 backtrace 출력은 다음과 비슷합니다.

0x0000000800426eff: debug_panic (lib/debug.c:86)
0x00000008004202fb: file_seek (filesys/file.c:405)
0x000000080042dc22: seek (userprog/syscall.c:744)
0x000000080042cf67: syscall_handler (userprog/syscall.c:444)
0x0000000800422319: intr_handler (threads/interrupt.c:334)
0x000000080042325a: intr_entry (threads/intr-stubs.S:38)
0x000000000040012c: (unknown)
0x0000000000400a96: (unknown)
0x0000000000400ac8: (unknown)

자기 kernel binary에서 위 command를 실행하면 주소가 정확히 같게 나오지는 않을 가능성이 큽니다. 컴파일한 source code와 compiler가 예제와 다를 수 있기 때문입니다.

Backtrace의 첫 줄은 kernel panic을 구현하는 함수인 debug_panic()을 가리킵니다. backtrace는 흔히 kernel panic의 결과로 나오므로, debug_panic()이 첫 함수로 표시되는 경우가 많습니다. 두 번째 줄은 이 경우 assertion failure의 결과로 panic한 함수가 file_seek()임을 보여 줍니다. 이 예제의 source tree에서 filesys/file.c 405번째 줄은 다음 assertion입니다.

ASSERT (file_ofs >= 0);

이 줄은 assertion failure message에도 함께 인용됩니다. 따라서 file_seek()는 음수 file offset 인자를 받아 panic했습니다. 세 번째 줄은 seek()가 offset 인자를 검증하지 않은 채 file_seek()를 호출했을 가능성을 보여 줍니다. 이 제출물에서 seek()는 seek system call을 구현합니다. 네 번째 줄은 system call handler인 syscall_handler()가 seek()를 호출했음을 보여 줍니다. 다섯 번째와 여섯 번째 줄은 interrupt handler entry path입니다. 나머지 줄들은 KERN_BASE보다 낮은 주소이므로 kernel이 아니라 user program 안의 주소를 뜻합니다.

GDB

Pintos는 GDB debugger의 감독 아래에서 실행할 수 있습니다. 먼저 pintos --gdb -- run mytest처럼 --gdb option으로 Pintos를 시작합니다. 그 다음 같은 머신의 두 번째 terminal을 열고, gdb로 kernel.o를 대상으로 GDB를 실행합니다.

gdb kernel.o

그리고 GDB에서 다음 command를 입력합니다.

target remote localhost:1234

이제 GDB가 local network connection을 통해 simulator에 연결되었습니다. 이후에는 일반적인 GDB command를 사용할 수 있습니다. c command를 입력하면 simulated BIOS가 제어를 가져가 Pintos를 load하고, Pintos가 보통 방식으로 실행됩니다. 언제든 Ctrl+C로 실행을 멈출 수 있습니다.

GDB 사용

Terminal command prompt에서 info gdb를 입력하면 GDB manual을 읽을 수 있습니다. 자주 유용한 GDB command는 다음과 같습니다.

  • c

    실행을 계속합니다. Ctrl+C가 눌리거나 다음 breakpoint에 도달할 때까지 계속됩니다.

  • break function

    지정한 function에 breakpoint를 설정합니다.

  • break file:line

    지정한 file의 line에 breakpoint를 설정합니다.

  • break address

    지정한 address에 breakpoint를 설정합니다. hexadecimal address를 지정하려면 0x prefix를 사용합니다. Pintos가 실행되기 시작하는 지점에서 멈추고 싶으면 break main을 사용하세요.

  • p expression

    주어진 expression을 평가하고 값을 출력합니다. expression 안에 function call이 들어 있으면 그 함수가 실제로 실행됩니다.

  • l address

    address 주변의 source code 몇 줄을 출력합니다. hexadecimal address를 지정하려면 0x prefix를 사용합니다.

  • bt

    위에서 설명한 backtrace program 출력과 비슷한 stack backtrace를 출력합니다.

  • p/a address

    address를 차지하는 함수나 변수의 이름을 출력합니다. hexadecimal address를 지정하려면 0x prefix를 사용합니다.

  • disassemble function

    function을 disassemble합니다.

추가 팁: GDB script인 pwndbg 같은 도구를 설치하면 디버깅이 훨씬 편해집니다.

FAQ

  • Emacs 안에서 GDB를 사용할 수 있나요?

    네. Emacs는 GDB를 subprocess로 실행하는 기능을 제공합니다. M-x gdb를 입력하고 prompt에 pintos-gdb command를 입력하세요. Emacs manual의 "Debuggers" 절에 디버깅 기능 사용법이 설명되어 있습니다.

  • GDB가 이상하게 동작합니다.

    GDB 사용 중 이상한 동작을 보면 세 가지 가능성이 있습니다. 수정한 Pintos의 bug, Bochs의 GDB interface 또는 GDB 자체의 bug, 원래 Pintos 코드의 bug입니다. 첫 번째와 두 번째 가능성이 꽤 높으므로 둘 다 진지하게 고려해야 합니다. 세 번째 가능성은 낮기를 바라지만, 불가능하지는 않습니다.

Triple Faults

Page fault handler 같은 CPU exception handler가 없거나 잘못되어 호출될 수 없으면, CPU는 "double fault" handler를 호출하려고 합니다. double fault handler 자체도 없거나 잘못되어 있으면 이것을 "triple fault"라고 부릅니다. Triple fault는 즉시 CPU reset을 일으킵니다.

따라서 머신이 계속 reboot loop에 빠지는 상황이라면 대개 triple fault일 가능성이 큽니다. triple fault 상황에서는 printf() 디버깅을 사용하지 못할 수도 있습니다. printf()에 필요한 모든 것이 초기화되기도 전에 reboot가 일어날 수 있기 때문입니다.

Triple fault를 디버깅하는 방법은 있습니다. Pintos 코드의 한 지점을 골라 그곳에 infinite loop인 for (;;);를 넣고, 다시 compile하고 실행합니다. 가능한 결과는 보통 두 가지입니다.

  • 머신이 reboot하지 않고 멈춥니다. 이 경우 infinite loop까지 실행된 것입니다. 따라서 reboot를 일으킨 원인은 infinite loop를 넣은 지점 뒤에 있습니다. 이제 infinite loop를 코드 흐름에서 더 뒤쪽으로 옮깁니다.

  • 머신이 계속 reboot loop에 빠집니다. 이 경우 머신이 infinite loop까지 도달하지 못한 것입니다. 따라서 reboot를 일으킨 원인은 infinite loop를 넣은 지점 앞에 있습니다. 이제 infinite loop를 코드 흐름에서 더 앞쪽으로 옮깁니다.

이 infinite loop 위치를 binary search하듯 옮기면, 모든 것이 잘못되기 시작하는 정확한 위치를 찾아낼 수 있습니다. 보통 길어도 몇 분이면 충분합니다.

팁

threads/palloc.c의 page allocator와 threads/malloc.c의 block allocator는 메모리를 free할 때 모든 byte를 0xcc로 채웁니다. 따라서 0xcccccccc 같은 pointer를 dereference하려 하거나 0xcc가 관련된 참조를 보게 된다면, 이미 free된 page를 다시 사용하려는 중일 가능성이 큽니다.

또한 byte 0xcc는 CPU opcode로 "interrupt 3 호출"을 뜻합니다. 그래서 Interrupt 0x03 (#BP Breakpoint Exception) 같은 오류가 보인다면, Pintos가 free된 page나 block 안의 코드를 실행하려 했다는 뜻일 수 있습니다. sec_no < d->capacity expression에서 assertion failure가 발생하면, Pintos가 닫히고 free된 inode를 통해 file에 접근하려 했음을 나타냅니다. inode가 free되면 시작 sector number가 0xcccccccc로 지워지는데, 약 1.6 TB보다 작은 disk에서는 유효한 sector number가 아닙니다.
