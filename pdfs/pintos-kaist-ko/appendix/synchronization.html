동기화

스레드들이 자원을 공유할 때 조심스럽고 통제된 방식으로 다루지 않으면 결과는 대개 크게 꼬입니다. 특히 운영체제 커널에서는 잘못된 공유가 전체 머신을 멈추게 할 수 있습니다. Pintos는 이런 문제를 다루기 위해 몇 가지 synchronization primitive를 제공합니다.

Interrupt 비활성화

가장 거친 동기화 방법은 interrupt를 끄는 것입니다. 즉 CPU가 interrupt에 일시적으로 반응하지 못하게 합니다. interrupt가 꺼져 있으면 running thread를 다른 스레드가 preempt할 수 없습니다. thread preemption은 timer interrupt에 의해 구동되기 때문입니다. 반대로 일반적인 경우처럼 interrupt가 켜져 있으면, 실행 중인 스레드는 두 C 문장 사이뿐 아니라 한 문장이 실행되는 도중에도 언제든 다른 스레드에게 선점될 수 있습니다.

이 말은 Pintos가 "preemptible kernel"이라는 뜻이기도 합니다. kernel thread가 언제든 선점될 수 있다는 의미입니다. 전통적인 Unix 시스템은 "nonpreemptible"입니다. 즉 kernel thread가 명시적으로 scheduler를 호출하는 지점에서만 선점될 수 있습니다. 두 모델 모두에서 user program은 언제든 선점될 수 있습니다. 예상할 수 있듯이 preemptible kernel은 더 명시적인 동기화를 필요로 합니다.

대부분의 경우 interrupt 상태를 직접 조작할 필요는 거의 없습니다. 보통은 뒤에서 설명하는 다른 synchronization primitive를 사용해야 합니다. interrupt를 끄는 주된 이유는 kernel thread와 external interrupt handler를 동기화해야 할 때입니다. external interrupt handler는 sleep할 수 없으므로 대부분의 다른 동기화 방식을 사용할 수 없습니다.

일부 external interrupt는 interrupt를 꺼도 미룰 수 없습니다. 이런 interrupt를 non-maskable interrupt(NMI)라고 부르며, 예컨대 컴퓨터가 불타는 것 같은 비상 상황에서만 쓰도록 되어 있습니다. Pintos는 non-maskable interrupt를 처리하지 않습니다.

interrupt를 끄고 켜는 데 필요한 타입과 함수는 include/threads/interrupt.h에 있습니다.

enum intr_level;

INTR_OFF 또는 INTR_ON 중 하나입니다. 각각 interrupt가 비활성화되었거나 활성화되었음을 뜻합니다.

enum intr_level intr_get_level (void)

현재 interrupt 상태를 반환합니다.

enum intr_level intr_set_level (enum intr_level level);

level에 따라 interrupt를 켜거나 끕니다. 이전 interrupt 상태를 반환합니다.

enum intr_level intr_enable (void);

interrupt를 켭니다. 이전 interrupt 상태를 반환합니다.

enum intr_level intr_disable (void);

interrupt를 끕니다. 이전 interrupt 상태를 반환합니다.

Semaphore

Semaphore는 음수가 아닌 정수 하나와, 그 정수를 원자적으로 조작하는 두 연산으로 구성됩니다.

  • "Down" 또는 "P": 값이 양수가 될 때까지 기다린 다음 값을 1 줄입니다.

  • "Up" 또는 "V": 값을 1 늘립니다. 기다리는 스레드가 있으면 그중 하나를 깨웁니다.

0으로 초기화한 semaphore는 정확히 한 번 일어날 사건을 기다리는 데 쓸 수 있습니다. 예를 들어 thread A가 thread B를 시작한 뒤, B가 어떤 작업을 끝냈다는 신호를 보낼 때까지 기다리고 싶다고 합시다. A는 0으로 초기화된 semaphore를 만들고, B를 시작할 때 그 semaphore를 넘긴 다음, semaphore에 대해 "down"을 수행합니다. B가 작업을 끝내면 semaphore에 대해 "up"을 수행합니다. A가 먼저 down을 하든 B가 먼저 up을 하든 이 방식은 올바르게 동작합니다.

struct semaphore sema;

/* Thread A */
void threadA (void) {
    sema_down (&sema);
}

/* Thread B */
void threadB (void) {
    sema_up (&sema);
}

/* main function */
void main (void) {
    sema_init (&sema, 0);
    thread_create ("threadA", PRI_MIN, threadA, NULL);
    thread_create ("threadB", PRI_MIN, threadB, NULL);
}

이 예에서 threadA는 threadB가 sema_up()을 호출할 때까지 sema_down()에서 실행을 멈춥니다.

1로 초기화한 semaphore는 보통 자원 접근을 제어하는 데 쓰입니다. 어떤 코드 블록이 자원을 사용하기 전에 semaphore를 down하고, 자원 사용이 끝나면 up합니다. 이런 경우에는 아래에서 설명하는 lock이 더 적절할 수 있습니다.

Semaphore는 1보다 큰 값으로 초기화할 수도 있지만, 이런 사용은 드뭅니다. Semaphore는 Edsger Dijkstra가 고안했고 THE operating system에서 처음 사용되었습니다. Pintos의 semaphore 타입과 연산은 include/threads/synch.h에 선언되어 있습니다.

struct semaphore;

Semaphore를 나타냅니다.

void sema_init (struct semaphore *sema, unsigned value);

sema를 주어진 초기값 value를 가진 새 semaphore로 초기화합니다.

void sema_down (struct semaphore *sema);

sema에 대해 "down" 또는 "P" 연산을 수행합니다. 값이 양수가 될 때까지 기다린 뒤 값을 1 줄입니다.

bool sema_try_down (struct semaphore *sema);

기다리지 않고 sema에 대해 "down" 또는 "P" 연산을 시도합니다. 성공적으로 값을 줄였으면 true를 반환하고, 값이 이미 0이라 기다리지 않고는 줄일 수 없으면 false를 반환합니다. 이 함수를 tight loop에서 반복 호출하면 CPU 시간을 낭비하므로, 대신 sema_down()을 쓰거나 다른 접근을 찾으세요.

void sema_up (struct semaphore *sema);

sema에 대해 "up" 또는 "V" 연산을 수행해 값을 1 늘립니다. sema에서 기다리는 스레드가 있으면 그중 하나를 깨웁니다. 대부분의 synchronization primitive와 달리 sema_up()은 external interrupt handler 안에서도 호출할 수 있습니다.

Semaphore는 내부적으로 interrupt 비활성화, 그리고 thread_block()과 thread_unblock()을 이용한 thread block/unblock으로 구현됩니다. 각 semaphore는 lib/kernel/list.c의 linked list 구현을 사용해 기다리는 스레드 목록을 유지합니다.

Lock

Lock은 초기값이 1인 semaphore와 비슷합니다. semaphore의 "up"에 해당하는 lock 연산은 "release"이고, "down"에 해당하는 연산은 "acquire"입니다.

Semaphore와 비교하면 lock에는 한 가지 추가 제약이 있습니다. lock을 acquire한 스레드, 즉 lock의 "owner"만 그 lock을 release할 수 있습니다. 이 제약이 문제가 된다면 lock 대신 semaphore를 써야 한다는 신호일 가능성이 큽니다.

Pintos의 lock은 recursive하지 않습니다. 즉 현재 어떤 lock을 보유 중인 스레드가 같은 lock을 다시 acquire하려고 하면 오류입니다. lock 타입과 함수는 include/threads/synch.h에 선언되어 있습니다.

struct lock;

Lock을 나타냅니다.

void lock_init (struct lock *lock);

lock을 새 lock으로 초기화합니다. 초기에는 어떤 스레드도 이 lock을 소유하지 않습니다.

void lock_acquire (struct lock *lock);

현재 스레드가 lock을 acquire합니다. 이미 owner가 있으면 필요에 따라 그 owner가 release할 때까지 기다립니다.

bool lock_try_acquire (struct lock *lock);

기다리지 않고 현재 스레드가 lock을 acquire하려고 시도합니다. 성공하면 true, lock이 이미 소유 중이면 false를 반환합니다. 이 함수를 tight loop에서 반복 호출하는 것은 CPU 시간을 낭비하므로 좋지 않습니다. 보통은 lock_acquire()를 쓰세요.

void lock_release (struct lock *lock);

lock을 release합니다. 현재 스레드가 이 lock을 소유하고 있어야 합니다.

bool lock_held_by_current_thread (const struct lock *lock):

실행 중인 스레드가 lock을 소유하고 있으면 true, 그렇지 않으면 false를 반환합니다. 임의의 다른 스레드가 lock을 소유하고 있는지 검사하는 함수는 없습니다. 검사 결과를 호출자가 사용하기 전에 답이 바뀔 수 있기 때문입니다.

Monitor

Monitor는 semaphore나 lock보다 높은 수준의 동기화 방식입니다. Monitor는 동기화 대상 데이터, monitor lock이라고 부르는 lock, 그리고 하나 이상의 condition variable로 구성됩니다. 스레드는 보호되는 데이터에 접근하기 전에 먼저 monitor lock을 acquire합니다. 그러면 그 스레드는 "monitor 안에 있다"고 말합니다. monitor 안에 있는 동안 스레드는 보호된 데이터를 자유롭게 검사하거나 수정할 수 있습니다. 접근이 끝나면 monitor lock을 release합니다.

Condition variable은 monitor 안의 코드가 어떤 조건이 true가 될 때까지 기다릴 수 있게 합니다. 각 condition variable은 추상적인 조건과 연결됩니다. 예를 들어 "처리할 데이터가 도착했다" 또는 "사용자의 마지막 키 입력 이후 10초가 넘었다" 같은 조건입니다. monitor 안의 코드가 어떤 조건이 true가 되기를 기다려야 하면, 관련 condition variable에서 "wait"합니다. 이 동작은 lock을 release하고 조건이 signal될 때까지 기다립니다. 반대로 어떤 코드가 그런 조건 중 하나를 true로 만들었다면, 그 condition에 "signal"을 보내 waiting thread 하나를 깨우거나, "broadcast"로 모두 깨울 수 있습니다.

Monitor의 이론적 틀은 C. A. R. Hoare가 제시했습니다. 실제 사용 방식은 이후 Mesa operating system 논문에서 더 구체화되었습니다. condition variable 타입과 함수는 include/threads/synch.h에 선언되어 있습니다.

struct condition;

Condition variable을 나타냅니다.

void cond_init (struct condition *cond);

cond를 새 condition variable로 초기화합니다.

void cond_wait (struct condition *cond, struct lock *lock);

lock, 즉 monitor lock을 원자적으로 release하고, 다른 코드가 cond에 signal을 보낼 때까지 기다립니다. cond가 signal된 뒤에는 반환하기 전에 lock을 다시 acquire합니다. 이 함수를 호출하기 전에는 반드시 lock을 보유하고 있어야 합니다.

Signal을 보내는 것과 wait에서 깨어나는 것은 하나의 원자적 연산이 아닙니다. 따라서 보통 cond_wait() 호출자는 wait가 끝난 뒤 조건을 다시 검사하고, 필요하면 다시 기다려야 합니다. 다음 절의 예제를 참고하세요.

void cond_signal (struct condition *cond, struct lock *lock);

cond에서 기다리는 스레드가 있으면 그중 하나를 깨웁니다. 이 condition은 monitor lock인 lock으로 보호됩니다. 기다리는 스레드가 없으면 아무 일도 하지 않고 반환합니다. 호출 전에 반드시 lock을 보유하고 있어야 합니다.

void cond_broadcast (struct condition *cond, struct lock *lock);

cond에서 기다리는 모든 스레드를 깨웁니다. 기다리는 스레드가 없으면 아무 일도 하지 않습니다. 이 condition은 monitor lock인 lock으로 보호되며, 호출 전에 반드시 lock을 보유하고 있어야 합니다.

Monitor 예제

Monitor의 고전적인 예는 하나 이상의 "producer" 스레드가 문자를 쓰고, 하나 이상의 "consumer" 스레드가 문자를 읽는 buffer를 다루는 것입니다. 이를 구현하려면 monitor lock 외에 condition variable 두 개가 필요합니다. 여기서는 not_full과 not_empty라고 부릅니다.

    char buf[BUF_SIZE];     /* Buffer. */
    size_t n = 0;         /* 0 <= n <= BUF SIZE: # of characters in buffer. */
    size_t head = 0;        /* buf index of next char to write (mod BUF SIZE). */
    size_t tail = 0;         /* buf index of next char to read (mod BUF SIZE). */
    struct lock lock;         /* Monitor lock. */
    struct condition not_empty; /* Signaled when the buffer is not empty. */
    struct condition not_full;     /* Signaled when the buffer is not full. */

    ...initialize the locks and condition variables...

    void put (char ch) {
      lock_acquire (&lock);
      while (n == BUF_SIZE)    /* Can't add to buf as long as it's full. */
        cond_wait (&not_full, &lock);
      buf[head++ % BUF_SIZE] = ch;    /* Add ch to buf. */
      n++;
      cond_signal (&not_empty, &lock);    /* buf can't be empty anymore. */
      lock_release (&lock);
    }

    char get (void) {
      char ch;
      lock_acquire (&lock);
      while (n == 0)        /* Can't read buf as long as it's empty. */
        cond_wait (&not_empty, &lock);
      ch = buf[tail++ % BUF_SIZE];    /* Get ch from buf. */
      n--;
      cond_signal (&not_full, &lock);    /* buf can't be full anymore. */
      lock_release (&lock);
    }

위 코드가 완전히 올바르려면 BUF_SIZE가 SIZE_MAX + 1을 나누어떨어뜨려야 합니다. 그렇지 않으면 head가 처음으로 0으로 wrap around될 때 실패합니다. 실제로는 BUF_SIZE를 보통 2의 거듭제곱으로 둡니다.

Optimization Barrier

Optimization barrier는 compiler가 barrier 앞뒤의 메모리 상태에 대해 특정 가정을 하지 못하게 하는 특수한 문장입니다. 주소를 한 번도 취하지 않은 local variable을 제외하면, compiler는 barrier를 가로질러 변수의 read/write 순서를 바꾸지 않고, barrier 앞뒤에서 변수 값이 바뀌지 않았다고 가정하지도 않습니다. Pintos에서는 include/threads/synch.h가 barrier() macro를 optimization barrier로 정의합니다.

Optimization barrier를 사용하는 한 가지 이유는 compiler가 모르는 사이에 데이터가 비동기적으로 바뀔 수 있을 때입니다. 예를 들어 다른 스레드나 interrupt handler가 값을 바꿀 수 있습니다. devices/timer.c의 too_many_loops()가 그런 예입니다. 이 함수는 timer tick이 발생할 때까지 loop에서 busy-wait하며 시작합니다.

    /* Wait for a timer tick. */
    int64_t start = ticks;
    while (ticks == start)
        barrier();

loop 안에 optimization barrier가 없으면 compiler는 이 loop가 끝나지 않는다고 결론낼 수 있습니다. start와 ticks는 처음에 같고, loop 자체는 둘 중 어느 값도 바꾸지 않기 때문입니다. 그러면 compiler가 함수를 infinite loop로 "최적화"할 수 있는데, 이는 명백히 원하지 않는 결과입니다.

Optimization barrier는 다른 compiler optimization을 막는 데도 사용할 수 있습니다. 역시 devices/timer.c에 있는 busy_wait() 함수가 예입니다. 이 함수에는 다음 loop가 있습니다.

    while (loops-- > 0)
      barrier ();

이 loop의 목적은 loops를 원래 값에서 0까지 세며 busy-wait하는 것입니다. barrier가 없으면 이 loop는 유용한 출력을 만들지 않고 side effect도 없으므로 compiler가 loop 전체를 삭제할 수 있습니다. barrier는 compiler가 loop body에 중요한 효과가 있다고 가정하도록 만듭니다.

마지막으로 optimization barrier는 memory read/write 순서를 강제하는 데도 쓸 수 있습니다. 예를 들어 timer interrupt가 발생할 때마다 global variable timer_put_char에 들어 있는 문자를 console에 출력하되, global Boolean variable timer_do_put이 true일 때만 출력하는 "기능"을 추가한다고 합시다. 문자 x가 출력되도록 준비하는 가장 좋은 방법은 다음처럼 optimization barrier를 사용하는 것입니다.

    timer_put_char = 'x';
    barrier ();
    timer_do_put = true;

barrier가 없으면 이 코드는 버그가 있습니다. compiler는 같은 순서를 유지해야 할 이유를 알지 못하면 연산 순서를 바꿀 수 있기 때문입니다. 이 경우 compiler는 두 assignment의 순서가 중요하다는 사실을 모르므로 optimizer가 순서를 바꿔도 됩니다. 실제로 순서를 바꿀지는 알 수 없고, compiler optimization flag나 compiler 버전에 따라 동작이 달라질 수도 있습니다.

다른 해결책은 assignment 주변에서 interrupt를 끄는 것입니다. 이것은 compiler reordering 자체를 막지는 않지만, 두 assignment 사이에 interrupt handler가 끼어드는 것을 막습니다. 대신 interrupt를 끄고 다시 켜는 runtime cost가 추가됩니다.

    enum intr_level old_level = intr_disable ();
    timer_put_char = 'x';
    timer_do_put = true;
    intr_set_level (old_level);

두 번째 해결책은 timer_put_char와 timer_do_put 선언에 volatile을 붙이는 것입니다. 이 keyword는 compiler에게 해당 변수가 외부에서 관찰될 수 있음을 알려 최적화 자유도를 제한합니다. 하지만 volatile의 의미는 잘 정의되어 있지 않으므로 일반적인 해결책으로 좋지 않습니다. 기본 Pintos 코드는 volatile을 전혀 사용하지 않습니다.

다음은 해결책이 아닙니다. lock은 interrupt를 막지 않고, lock이 잡혀 있는 영역 안에서 compiler가 코드 순서를 바꾸는 것도 막지 않기 때문입니다.

    lock_acquire (&timer_lock);        /* INCORRECT CODE */
    timer_put_char = 'x';
    timer_do_put = true;
    lock_release (&timer_lock);

Compiler는 다른 source file에 정의된 외부 함수 호출을 제한적인 형태의 optimization barrier처럼 취급합니다. 구체적으로 compiler는 외부 정의 함수가 정적 또는 동적으로 할당된 어떤 데이터든 접근할 수 있고, 주소가 취해진 local variable에도 접근할 수 있다고 가정합니다. 그래서 명시적인 barrier를 생략할 수 있는 경우가 많습니다. Pintos 코드에 명시적인 barrier가 적은 이유 중 하나입니다.

반대로 같은 source file에 정의된 함수나 그 source file이 include한 header에 정의된 함수는 optimization barrier로 믿을 수 없습니다. 함수 정의보다 앞에서 호출하더라도 마찬가지입니다. compiler는 optimization을 수행하기 전에 전체 source file을 읽고 parse할 수 있기 때문입니다.
