원문 보기
KAIST Pintos GitBook 원문을 한국어로 옮긴 로컬 읽기본입니다. 코드, 명령어, 함수명, 파일명은 의도적으로 원문 형태를 최대한 유지했습니다. 의미가 애매한 부분은 원문 링크와 함께 확인하세요.

해시 테이블

Pintos는 lib/kernel/hash.c에 hash table 자료구조를 제공합니다. 사용하려면 header file인 lib/kernel/hash.h를 include해야 하며, 보통 #include <hash.h>를 사용합니다. Pintos가 제공하는 기본 코드는 hash table을 사용하지 않습니다. 따라서 필요에 따라 그대로 쓰거나, 목적에 맞게 구현을 바꾸거나, 아예 사용하지 않아도 됩니다.

Virtual memory project 구현의 상당수는 page에서 frame으로 변환하기 위해 hash table을 사용합니다. 그 밖에도 hash table을 쓸 만한 곳을 찾을 수 있습니다.

데이터 타입

Hash table 전체는 struct hash로 표현됩니다.

struct hash;

하나의 hash table 전체를 나타냅니다. struct hash의 실제 멤버는 opaque입니다. 즉 hash table을 사용하는 코드는 struct hash 멤버에 직접 접근하면 안 되고, 그럴 필요도 없습니다. 대신 hash table 함수와 macro를 사용하세요.

Hash table은 struct hash_elem 타입의 element를 다룹니다.

struct hash_elem;

Hash table에 넣고 싶은 구조체 안에 struct hash_elem 멤버를 포함시키세요. struct hash와 마찬가지로 struct hash_elem도 opaque입니다. Hash table element를 다루는 모든 함수는 실제 element 타입의 포인터가 아니라 struct hash_elem 포인터를 받고 반환합니다.

실제 hash table element에서 그 안의 struct hash_elem을 얻어야 할 때가 있고, 반대로 struct hash_elem에서 실제 element 구조체를 얻어야 할 때도 많습니다. 실제 element가 있으면 & 연산자로 그 안의 struct hash_elem 포인터를 얻을 수 있습니다. 반대 방향으로는 hash_entry() macro를 사용합니다.

#define hash_entry (elem, type, member) { /* Omit details */ }

struct hash_elem 포인터인 elem이 포함되어 있는 바깥 구조체의 포인터를 반환합니다. type에는 elem이 들어 있는 구조체 타입 이름을, member에는 그 타입 안에서 elem이 가리키는 멤버 이름을 넘겨야 합니다.

예를 들어 h가 struct hash_elem * 변수이고, struct thread 안의 h_elem이라는 struct hash_elem 멤버를 가리킨다고 합시다. 그러면 hash_entry (h, struct thread, h_elem)은 h가 포함되어 있는 struct thread의 주소를 돌려줍니다.

각 hash table element는 key를 가져야 합니다. key는 element를 식별하고 서로 구분하는 데이터이며, 같은 hash table 안에서 유일해야 합니다. element는 key가 아닌 데이터를 함께 가질 수도 있고, 그런 non-key data는 유일할 필요가 없습니다. element가 hash table 안에 들어 있는 동안에는 key data를 바꾸면 안 됩니다. key를 바꿔야 한다면 element를 hash table에서 제거하고, key를 수정한 뒤 다시 삽입하세요.

각 hash table에 대해 key를 다루는 함수 두 개를 작성해야 합니다. 하나는 hash function이고, 다른 하나는 comparison function입니다. 이 함수들은 다음 prototype과 맞아야 합니다.

typedef unsigned hash_hash_func (const struct hash_elem *e, void *aux);

element data의 hash 값을 unsigned int 범위 안의 값으로 반환합니다. element의 hash는 element key의 pseudo-random function이어야 합니다. element 안의 non-key data나 key가 아닌 비상수 데이터에 의존해서는 안 됩니다. Pintos는 hash function의 기반으로 쓰기 좋은 다음 함수들을 제공합니다.

unsigned hash_bytes (const void *buf, size_t *size)

buf에서 시작하는 size byte의 hash를 반환합니다. 구현은 32-bit word용 general-purpose Fowler-Noll-Vo hash입니다.

unsigned hash_string (const char *s)

null-terminated string s의 hash를 반환합니다.

unsigned hash_int (int i)

integer i의 hash를 반환합니다.

key가 적절한 타입의 단일 데이터라면, 위 함수 중 하나의 출력을 그대로 hash function 결과로 반환하는 것이 자연스럽습니다. 여러 데이터 조각으로 된 key라면, 예를 들어 ^ exclusive-or 연산자를 사용해 여러 호출 결과를 조합할 수 있습니다. 물론 이 함수들을 모두 무시하고 직접 hash function을 작성할 수도 있습니다. 다만 목표는 운영체제 kernel을 만드는 것이지 hash function을 설계하는 것이 아니라는 점을 기억하세요. aux에 대한 설명은 Auxiliary Data를 참고하세요.

bool hash_less_func (const struct hash_elem *a,
                     const struct hash_elem *b,
                     void *aux)

element a와 b에 저장된 key를 비교합니다. a가 b보다 작으면 true, a가 b보다 크거나 같으면 false를 반환합니다. 두 element가 같다고 비교된다면, 둘은 반드시 같은 hash 값으로 hash되어야 합니다. aux는 Auxiliary Data에서 설명합니다.

void hash_action_func (struct hash_elem *element, void *aux)

호출자가 선택한 어떤 동작을 element에 대해 수행합니다. aux는 Auxiliary Data에서 설명합니다.

Hash function과 comparison function의 예는 Hash Table 예제를 참고하세요.

기본 함수

다음 함수들은 hash table을 생성, 파괴, 검사합니다.

bool hash_init (struct hash *hash, hash_hash_func *hash_func,
                hash_less_func *less_func, void *aux)

hash를 새 hash table로 초기화합니다. hash_func는 hash function, less_func는 comparison function, aux는 auxiliary data입니다. 성공하면 true, 실패하면 false를 반환합니다. hash_init()은 malloc()을 호출하므로 메모리를 할당할 수 없으면 실패합니다. aux는 보통 null pointer이며, 자세한 내용은 Auxiliary Data를 참고하세요.

void hash_clear (struct hash *hash, hash_action_func *action)

이전에 hash_init()으로 초기화된 hash에서 모든 element를 제거합니다. action이 null이 아니면 hash table의 각 element마다 한 번씩 호출됩니다. 이를 통해 호출자는 element가 사용하던 메모리나 다른 자원을 해제할 기회를 얻습니다. 예를 들어 hash table element를 malloc()으로 동적 할당했다면, action에서 그 element를 free()할 수 있습니다. hash_clear()는 어떤 element에 대해 action을 호출한 뒤에는 그 element의 메모리에 다시 접근하지 않으므로 안전합니다. 하지만 action은 hash_insert()나 hash_delete()처럼 hash table을 수정할 수 있는 함수를 호출하면 안 됩니다.

void hash_destroy (struct hash *hash, hash_action_func *action);

action이 null이 아니면 hash_clear()와 같은 의미로 hash의 각 element에 대해 호출합니다. 그 다음 hash가 보유한 메모리를 해제합니다. 이후에는 다시 hash_init()을 호출하기 전까지 hash를 어떤 hash table 함수에도 넘기면 안 됩니다.

size_t hash_size (struct hash *hash);

현재 hash에 저장된 element 수를 반환합니다.

bool hash_empty (struct hash *hash);

hash가 현재 아무 element도 담고 있지 않으면 true, 하나 이상 담고 있으면 false를 반환합니다.

검색 함수

다음 함수들은 주어진 element와 같다고 비교되는 element를 hash table에서 찾습니다. 검색 성공 여부에 따라 새 element를 삽입하거나, 검색 결과를 그대로 반환하는 등의 동작을 수행합니다.

struct hash_elem *hash_insert (struct hash *hash,
                               struct hash_elem *element);

hash에서 element와 같은 element를 찾습니다. 없으면 element를 hash에 삽입하고 null pointer를 반환합니다. 이미 같은 element가 들어 있으면 hash를 수정하지 않고 기존 element를 반환합니다.

struct hash_elem *hash_replace (struct hash *hash,
                                struct hash_elem *element);

element를 hash에 삽입합니다. hash 안에 element와 같은 element가 이미 있으면 제거됩니다. 제거된 element를 반환하며, 같은 element가 없었다면 null pointer를 반환합니다.

반환된 element와 관련된 자원을 적절히 해제하는 책임은 호출자에게 있습니다. 예를 들어 hash table element가 malloc()으로 동적 할당된 것이라면, 더 이상 필요 없어진 뒤 호출자가 그 element를 free()해야 합니다.

다음 함수들에 넘기는 element는 hash와 comparison 목적으로만 사용됩니다. 실제로 hash table에 삽입되지는 않습니다. 따라서 이 element에서는 key data만 초기화하면 되고, 다른 데이터는 사용되지 않습니다. 흔한 방식은 element 타입의 local variable을 선언하고 key data만 초기화한 뒤, 그 안의 struct hash_elem 주소를 hash_find()나 hash_delete()에 넘기는 것입니다. 예시는 Hash Table 예제를 참고하세요. 큰 구조체를 local variable로 할당하면 안 된다는 점도 기억하세요. 자세한 내용은 struct thread를 참고하세요.

struct hash_elem *hash_find (struct hash *hash,
                             struct hash_elem *element);

hash에서 element와 같은 element를 찾습니다. 찾으면 그 element를 반환하고, 없으면 null pointer를 반환합니다.

struct hash_elem *hash_delete (struct hash *hash,
                               struct hash_elem *element);

hash에서 element와 같은 element를 찾습니다. 찾으면 hash table에서 제거하고 반환합니다. 없으면 null pointer를 반환하고 hash는 변경되지 않습니다.

반환된 element와 관련된 자원을 적절히 해제하는 책임은 호출자에게 있습니다. 예를 들어 hash table element가 malloc()으로 동적 할당된 것이라면, 더 이상 필요 없어진 뒤 호출자가 그 element를 free()해야 합니다.

순회 함수

다음 함수들은 hash table 안의 element들을 순회할 수 있게 합니다. 두 가지 interface가 제공됩니다. 첫 번째는 각 element에 대해 동작할 hash_action_func를 작성해 넘기는 방식입니다.

void hash_apply (struct hash *hash, hash_action_func *action);

hash 안의 각 element에 대해 임의의 순서로 action을 한 번씩 호출합니다. action은 hash_insert()나 hash_delete()처럼 hash table을 수정할 수 있는 함수를 호출하면 안 됩니다. 또한 element의 key data를 수정하면 안 됩니다. 단, key가 아닌 다른 데이터는 수정할 수 있습니다.

두 번째 interface는 "iterator" 데이터 타입을 기반으로 합니다. 보통 iterator는 다음처럼 사용합니다.

struct hash_iterator i;
hash_first (&i, h);
while (hash_next (&i)) {
    struct foo *f = hash_entry (hash_cur (&i), struct foo, elem);
    . . . do something with f . . .
}

struct hash_iterator;

Hash table 안의 한 위치를 나타냅니다. hash_insert()나 hash_delete()처럼 hash table을 수정할 수 있는 함수를 호출하면, 그 hash table에 대한 모든 iterator가 invalid해집니다. struct hash와 struct hash_elem처럼 struct hash_iterator도 opaque입니다.

void hash_first (struct hash_iterator *iterator, struct hash *hash);

iterator를 hash의 첫 element 바로 앞 위치로 초기화합니다.

struct hash_elem *hash_next (struct hash_iterator *iterator);

iterator를 hash의 다음 element로 진행시키고, 그 element를 반환합니다. 남은 element가 없으면 null pointer를 반환합니다. 어떤 iterator에 대해 hash_next()가 null을 반환한 뒤 다시 hash_next()를 호출하면 undefined behavior입니다.

struct hash_elem *hash_cur (struct hash_iterator *iterator);

iterator에 대해 가장 최근에 hash_next()가 반환한 값을 반환합니다. hash_first()를 호출한 뒤 아직 첫 번째 hash_next()를 호출하기 전에는 undefined behavior입니다.

Hash Table 예제

struct page라는 구조체를 hash table에 넣고 싶다고 합시다. 먼저 struct page 안에 struct hash_elem 멤버를 포함하도록 정의합니다.

struct page {
    struct hash_elem hash_elem; /* Hash table element. */
    void *addr; /* Virtual address. */
    /* . . . other members. . . */
};

addr를 key로 사용해 hash function과 comparison function을 작성합니다. Pointer는 그 byte들을 기반으로 hash할 수 있고, pointer 비교에는 < 연산자가 잘 동작합니다.

/* Returns a hash value for page p. */
unsigned
page_hash (const struct hash_elem *p_, void *aux UNUSED) {
  const struct page *p = hash_entry (p_, struct page, hash_elem);
  return hash_bytes (&p->addr, sizeof p->addr);
}

/* Returns true if page a precedes page b. */
bool
page_less (const struct hash_elem *a_,
           const struct hash_elem *b_, void *aux UNUSED) {
  const struct page *a = hash_entry (a_, struct page, hash_elem);
  const struct page *b = hash_entry (b_, struct page, hash_elem);

  return a->addr < b->addr;
}

이 함수 prototype에서 UNUSED를 사용한 것은 aux가 사용되지 않는다는 warning을 억제하기 위해서입니다. UNUSED에 대해서는 Function과 Parameter Attribute를 참고하세요. aux는 Auxiliary Data에서 설명합니다.

그 다음 hash table을 다음처럼 만들 수 있습니다.

struct hash pages;
hash_init (&pages, page_hash, page_less, NULL);

이제 만든 hash table을 조작할 수 있습니다. p가 struct page 포인터라면, 다음처럼 hash table에 삽입합니다.

hash_insert (&pages, &p->hash_elem);

pages가 같은 addr를 가진 page를 이미 포함할 가능성이 있다면, hash_insert()의 반환값을 확인해야 합니다.

Hash table에서 element를 찾으려면 hash_find()를 사용합니다. hash_find()는 비교 대상 element를 요구하므로 약간의 준비가 필요합니다. 다음 함수는 pages가 file scope에 정의되어 있다고 가정하고, virtual address를 기준으로 page를 찾아 반환합니다.

/* Returns the page containing the given virtual address, or a null pointer if no such page exists. */
struct page *
page_lookup (const void *address) {
  struct page p;
  struct hash_elem *e;

  p.addr = address;
  e = hash_find (&pages, &p.hash_elem);
  return e != NULL ? hash_entry (e, struct page, hash_elem) : NULL;
}

여기서는 struct page가 꽤 작다는 가정 아래 local variable로 할당했습니다. 큰 구조체는 local variable로 할당하면 안 됩니다. 비슷한 함수로 hash_delete()를 사용해 주소 기준 page 삭제도 구현할 수 있습니다.

Auxiliary Data

위 예제처럼 단순한 경우에는 aux parameter가 필요 없습니다. 이런 경우에는 hash_init()의 aux로 null pointer를 넘기고, hash function과 comparison function에 전달된 값을 무시하면 됩니다. aux parameter를 사용하지 않으면 compiler warning이 나올 수 있지만, 예제처럼 UNUSED macro로 끌 수 있고, 단순히 무시할 수도 있습니다.

aux는 hash table 안의 데이터에 대해 hashing이나 comparison에 필요한 어떤 속성이 constant이지만, data item 자체에는 저장되어 있지 않을 때 유용합니다. 예를 들어 hash table item들이 fixed-length string인데 item 자체에는 그 fixed length가 들어 있지 않다면, 길이를 aux parameter로 전달할 수 있습니다.

동기화

Hash table은 내부 synchronization을 수행하지 않습니다. Hash table 함수 호출을 동기화하는 책임은 호출자에게 있습니다. 일반적으로 hash_find()나 hash_next()처럼 hash table을 검사하지만 수정하지 않는 함수들은 여러 개가 동시에 실행될 수 있습니다. 그러나 이런 함수들은 hash_insert()나 hash_delete()처럼 해당 hash table을 수정할 수 있는 함수와 동시에 안전하게 실행될 수 없습니다. 또한 같은 hash table을 수정할 수 있는 함수가 둘 이상 동시에 실행되어서도 안 됩니다.

Hash table element 안의 데이터 접근을 동기화하는 것도 호출자의 책임입니다. 이 데이터 접근을 어떻게 동기화할지는 다른 자료구조와 마찬가지로 데이터가 어떻게 설계되고 구성되어 있는지에 달려 있습니다.
