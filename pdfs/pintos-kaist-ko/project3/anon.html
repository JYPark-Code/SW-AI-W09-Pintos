Anonymous Page

이 부분에서는 디스크 기반 이미지가 없는 page, 즉 anonymous page를 구현합니다.

anonymous mapping에는 backing file이나 device가 없습니다. file-backed page와 달리 이름이 붙은 파일 출처가 없기 때문에 anonymous라고 부릅니다. anonymous page는 stack이나 heap처럼 실행 파일에서 사용됩니다.

anonymous page를 설명하기 위한 구조체로 include/vm/anon.h의 anon_page가 있습니다. 현재는 비어 있지만, 구현하면서 anonymous page에 필요한 정보나 상태를 저장할 멤버를 추가할 수 있습니다. 또한 page의 공통 정보를 담는 include/vm/page.h의 struct page도 참고하세요. anonymous page의 경우 page 구조체 안에 struct anon_page anon이 포함됩니다.

Lazy Loading을 이용한 Page 초기화

lazy loading은 메모리 로딩을 실제로 필요해지는 시점까지 미루는 설계입니다. page는 할당됩니다. 즉 그에 대응하는 struct page는 존재합니다. 하지만 아직 전용 physical frame은 없고, page의 실제 내용도 로드되지 않았습니다. 내용은 정말 필요해졌을 때만 로드되며, 그 시점은 page fault로 알려집니다.

page 타입이 세 가지이므로 초기화 루틴도 page 타입마다 다릅니다. 아래 절에서 다시 설명하겠지만, 여기서는 page 초기화 흐름을 큰 그림으로 설명합니다. 커널이 새 page 요청을 받으면 먼저 vm_alloc_page_with_initializer가 호출됩니다. initializer는 새 page 구조체를 할당하고 page 타입에 맞는 initializer를 설정한 뒤 사용자 프로그램으로 제어를 돌려줍니다. 사용자 프로그램이 실행되다가 어느 시점에 아직 내용이 없는 page에 접근하면 page fault가 발생합니다. fault 처리 과정에서 uninit_initialize가 호출되고, 앞서 설정해 둔 initializer를 호출합니다. anonymous page의 initializer는 anon_initializer이고, file-backed page의 initializer는 file_backed_initializer입니다.

page의 생명주기는 대략 initialize -> (page_fault -> lazy-load -> swap-in -> swap-out -> ...) -> destroy 형태가 될 수 있습니다. 생명주기 전환마다 필요한 절차는 page 타입, 즉 VM_TYPE에 따라 다릅니다. 앞 문단은 초기화 과정의 예입니다. 이번 프로젝트에서는 각 page 타입에 대해 이런 전환 과정을 구현합니다.

실행 파일을 위한 Lazy Loading

lazy loading에서는 프로세스 실행이 시작될 때 당장 필요한 메모리 부분만 주 메모리에 올립니다. 바이너리 이미지 전체를 한 번에 메모리에 올리는 eager loading과 비교하면 오버헤드를 줄일 수 있습니다.

lazy loading을 지원하기 위해 include/vm/vm.h에 VM_UNINIT이라는 page 타입을 도입합니다. 모든 page는 처음에 VM_UNINIT page로 생성됩니다. 초기화되지 않은 page를 위한 page 구조체인 struct uninit_page도 include/vm/uninit.h에 제공됩니다. uninitialized page를 만들고, 초기화하고, 파괴하는 함수들은 include/vm/uninit.c에서 찾을 수 있습니다. 나중에 이 함수들을 완성해야 합니다.

page fault가 발생하면 userprog/exception.c의 page fault handler인 page_fault가 vm/vm.c의 vm_try_handle_fault로 제어를 넘깁니다. 이 함수는 먼저 처리 가능한 page fault인지 확인합니다. 처리할 수 없는 잘못된 fault라면 프로세스를 종료해야 합니다. 처리 가능한 fault라면 page에 필요한 내용을 로드한 뒤 사용자 프로그램으로 제어를 돌려줍니다.

처리 가능한 page fault에는 lazy-loaded page, swapped-out page, write-protected page가 있습니다. write-protected page는 Copy-on-Write (추가 과제)에서 다룹니다. 지금은 첫 번째 경우, lazy-loaded page만 생각하면 됩니다. lazy loading을 위한 page fault라면 커널은 앞서 vm_alloc_page_with_initializer에서 설정한 initializer 중 하나를 호출해 segment를 lazy load합니다. userprog/process.c의 lazy_load_segment를 구현해야 합니다.

vm_alloc_page_with_initializer()를 구현하세요. 전달된 vm_type에 따라 적절한 initializer를 가져오고, 그것과 함께 uninit_new를 호출해야 합니다.

bool vm_alloc_page_with_initializer (enum vm_type type, void *va,
        bool writable, vm_initializer *init, void *aux);

주어진 타입의 uninitialized page를 만듭니다. uninit page의 swap_in handler는 타입에 따라 page를 자동으로 초기화하고, 주어진 AUX와 함께 INIT을 호출합니다. page 구조체를 얻은 뒤에는 그 page를 프로세스의 supplemental page table에 삽입하세요. vm.h에 정의된 VM_TYPE 매크로를 사용하면 편리할 수 있습니다.

page fault handler는 호출 체인을 따라가고, swap_in을 호출했을 때 최종적으로 uninit_initialize에 도달합니다. 이 함수의 완성된 구현은 제공되어 있습니다. 다만 설계에 따라 uninit_initialize를 수정해야 할 수도 있습니다.

static bool uninit_initialize (struct page *page, void *kva);

첫 fault에서 page를 초기화합니다. 템플릿 코드는 먼저 vm_initializer와 aux를 가져오고, 함수 포인터를 통해 해당 page_initializer를 호출합니다. 설계에 따라 이 함수를 수정해야 할 수 있습니다.

필요에 따라 vm/anon.c의 vm_anon_init과 anon_initializer를 수정할 수 있습니다.

void vm_anon_init (void);

anonymous page subsystem을 초기화합니다. 이 함수 안에서 anonymous page와 관련된 어떤 설정이든 할 수 있습니다.

bool anon_initializer (struct page *page,enum vm_type type, void *kva);

이 함수는 먼저 page->operations에 anonymous page용 handler들을 설정합니다. 현재 비어 있는 anon_page에 필요한 정보를 갱신해야 할 수도 있습니다. 이 함수는 anonymous page, 즉 VM_ANON의 initializer로 사용됩니다.

userprog/process.c에 load_segment와 lazy_load_segment를 구현하세요. 실행 파일에서 segment를 로드하는 기능을 구현합니다. 이 모든 page는 lazy하게 로드되어야 합니다. 즉 커널이 해당 page에 대한 page fault를 가로챘을 때만 로드되어야 합니다.

프로그램 로더의 핵심인 userprog/process.c의 load_segment 안쪽 loop를 수정해야 합니다. loop를 한 번 돌 때마다 vm_alloc_page_with_initializer를 호출해서 pending page object를 만듭니다. 실제 segment가 파일에서 로드되는 시점은 page fault가 발생했을 때입니다.

static bool load_segment (struct file *file, off_t ofs, uint8_t *upage,
        uint32_t read_bytes, uint32_t zero_bytes, bool writable);

현재 코드는 main loop 안에서 파일에서 읽을 바이트 수와 0으로 채울 바이트 수를 계산합니다. 그 다음 vm_alloc_page_with_initializer를 호출해서 pending object를 만듭니다. vm_alloc_page_with_initializer에 넘길 aux 인자로 보조 값을 설정해야 합니다. 바이너리 로딩에 필요한 정보를 담는 구조체를 하나 만드는 것이 좋을 수 있습니다.

static bool lazy_load_segment (struct page *page, void *aux);

load_segment에서 vm_alloc_page_with_initializer의 네 번째 인자로 lazy_load_segment가 전달되는 것을 봤을 것입니다. 이 함수는 실행 파일 page의 initializer이며 page fault가 발생했을 때 호출됩니다. 인자로 struct page와 aux를 받습니다. aux는 load_segment에서 설정한 정보입니다. 이 정보를 사용해 segment를 읽어 올 파일을 찾고, 최종적으로 segment를 메모리에 읽어 와야 합니다.

userprog/process.c의 setup_stack도 새 메모리 관리 시스템에 맞게 조정해야 합니다. 첫 번째 stack page는 lazy하게 할당할 필요가 없습니다. load 시점에 command line arguments와 함께 할당하고 초기화하면 되며, fault가 날 때까지 기다릴 필요가 없습니다. stack을 식별할 방법이 필요할 수 있습니다. vm/vm.h의 vm_type 보조 marker, 예를 들어 VM_MARKER_0를 사용해서 page를 표시할 수 있습니다.

마지막으로 vm_try_handle_fault 함수를 수정해서 fault가 난 주소에 대응하는 page 구조체를 supplemental page table에서 spt_find_page로 찾고, 그 fault를 해결하도록 만드세요.

위 요구사항을 모두 구현하면 fork를 제외한 프로젝트 2의 모든 테스트가 통과해야 합니다.

Supplemental Page Table 다시 보기

이제 copy와 clean up 연산을 지원하기 위해 supplemental page table interface를 다시 살펴봅니다. 이 연산들은 프로세스를 만들 때, 더 구체적으로는 child process를 생성할 때와 프로세스를 파괴할 때 필요합니다. 자세한 내용은 아래에 설명합니다. 이 시점에서 supplemental page table을 다시 다루는 이유는 앞에서 구현한 초기화 함수들을 여기서 사용할 수 있기 때문입니다.

vm/vm.c에 supplemental_page_table_copy와 supplemental_page_table_kill을 구현하세요.

bool supplemental_page_table_copy (struct supplemental_page_table *dst,
    struct supplemental_page_table *src);

src의 supplemental page table을 dst로 복사합니다. 자식이 부모의 실행 context를 상속해야 할 때, 즉 fork()에서 사용됩니다. src의 supplemental page table 안에 있는 각 page를 순회하면서 dst의 supplemental page table에 정확한 복사본 entry를 만들어야 합니다. uninit page를 할당하고 곧바로 claim해야 합니다.

void supplemental_page_table_kill (struct supplemental_page_table *spt);

supplemental page table이 보유하던 모든 자원을 해제합니다. 이 함수는 프로세스가 exit할 때(userprog/process.c의 process_exit()) 호출됩니다. page entry들을 순회하면서 table 안의 page들에 대해 destroy(page)를 호출해야 합니다. 이 함수 안에서는 실제 page table(pml4)과 physical memory(palloc으로 할당된 메모리)를 걱정할 필요가 없습니다. 호출자가 supplemental page table 정리가 끝난 뒤 그것들을 정리합니다.

Page 정리

vm/uninit.c의 uninit_destroy와 vm/anon.c의 anon_destroy를 구현하세요. 이것들은 uninitialized page의 destroy operation을 위한 handler입니다. uninitialized page는 다른 page object로 변환되지만, 프로세스가 exit할 때 아직 uninit page가 남아 있을 수도 있습니다.

static void uninit_destroy (struct page *page);

page 구조체가 보유하던 자원을 해제합니다. page의 VM 타입을 확인해서 그에 맞게 처리하는 것이 좋을 수 있습니다.

지금은 anonymous page만 처리하면 됩니다. 나중에 file-backed page를 정리하기 위해 이 함수를 다시 살펴보게 됩니다.

static void anon_destroy (struct page *page);

anonymous page가 보유하던 자원을 해제합니다. page 구조체 자체를 명시적으로 해제할 필요는 없습니다. 호출자가 해제해야 합니다.
