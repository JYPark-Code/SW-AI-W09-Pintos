프로젝트 3: 가상 메모리

이제 Pintos 내부 동작에 어느 정도 익숙해졌을 것입니다. OS는 적절한 동기화와 함께 여러 실행 스레드를 처리할 수 있고, 여러 사용자 프로그램을 동시에 로드할 수도 있습니다. 하지만 실행 가능한 프로그램의 수와 크기는 여전히 머신의 주 메모리 크기에 제한됩니다. 이번 과제에서는 무한한 메모리가 있는 것처럼 보이게 만들어 그 제한을 제거합니다.

이번 과제는 이전 과제 위에 구현합니다. 프로젝트 2의 테스트 프로그램도 프로젝트 3에서 동작해야 합니다. 프로젝트 3을 시작하기 전에 프로젝트 2 제출물의 버그를 고쳐 두세요. 그 버그들은 프로젝트 3에서도 거의 같은 문제를 일으킬 가능성이 높습니다.

프로젝트 3에서는 구현을 쉽게 따라갈 수 있도록 단계별 지시를 제공합니다.

배경

소스 파일

이번 프로젝트에서는 vm 디렉터리에서 작업합니다. Makefile은 -DVM 설정이 켜지도록 갱신되어 있습니다. 많은 양의 템플릿 코드가 제공됩니다. 반드시 제공된 템플릿을 따라야 합니다. 제출한 코드가 제공된 템플릿 기반이 아니면 0점을 받습니다. 또한 "DO NOT CHANGE"라고 표시된 템플릿 부분은 절대 바꾸면 안 됩니다. 아래는 수정하게 될 템플릿 파일들의 역할입니다.

  • include/vm/vm.h, vm/vm.c

    가상 메모리에 대한 일반 인터페이스를 제공합니다. 헤더 파일에는 가상 메모리 시스템이 지원해야 하는 여러 vm_type, 즉 VM_UNINIT, VM_ANON, VM_FILE, VM_PAGE_CACHE의 정의와 설명이 있습니다. VM_PAGE_CACHE는 지금은 무시하세요. 여기에서 supplemental page table도 구현합니다.

  • include/vm/uninit.h, vm/uninit.c

    초기화되지 않은 페이지(vm_type = VM_UNINIT)에 대한 연산을 제공합니다. 현재 설계에서는 모든 페이지가 처음에는 uninitialized page로 설정되고, 이후 anonymous page나 file-backed page로 변환됩니다.

  • include/vm/anon.h, vm/anon.c

    anonymous page(vm_type = VM_ANON)에 대한 연산을 제공합니다.

  • include/vm/file.h, vm/file.c

    file-backed page(vm_type = VM_FILE)에 대한 연산을 제공합니다.

  • include/vm/inspect.h, vm/inspect.c

    채점을 위한 메모리 검사 연산이 들어 있습니다. 이 파일들은 수정하지 마세요.

이번 프로젝트에서 작성하는 코드 대부분은 vm 디렉터리의 파일들과 이전 프로젝트에서 소개된 파일들에 들어갑니다. 처음 보게 될 가능성이 있는 파일은 많지 않지만, 예를 들면 다음 파일이 있습니다.

  • include/devices/block.h, devices/block.c

    블록 장치에 섹터 단위로 읽고 쓰는 인터페이스를 제공합니다. swap partition을 블록 장치로 접근할 때 이 인터페이스를 사용합니다.

메모리 용어

먼저 메모리와 저장장치 관련 용어를 정리합니다. 일부 용어는 프로젝트 2의 가상 메모리 배치에서 이미 익숙할 수 있지만, 상당수는 새로 나옵니다.

페이지

page, 또는 virtual page는 크기가 4,096바이트인 연속된 가상 메모리 영역입니다. 이 크기를 page size라고 합니다. 페이지는 page-aligned되어야 합니다. 즉 가상 주소가 page size로 나누어떨어지는 위치에서 시작해야 합니다. 따라서 64비트 가상 주소의 마지막 12비트는 page offset, 또는 그냥 offset입니다. 위쪽 비트들은 곧 소개할 page table의 인덱스를 나타내는 데 사용됩니다. 64비트 시스템에서는 4-level page table을 사용하므로, 가상 주소는 다음과 같은 형태입니다.

63          48 47            39 38            30 29            21 20         12 11         0
+-------------+----------------+----------------+----------------+-------------+------------+
| Sign Extend |    Page-Map    | Page-Directory | Page-directory |  Page-Table |    Page    |
|             | Level-4 Offset |    Pointer     |     Offset     |   Offset    |   Offset   |
+-------------+----------------+----------------+----------------+-------------+------------+
              |                |                |                |             |            |
              +------- 9 ------+------- 9 ------+------- 9 ------+----- 9 -----+---- 12 ----+
                                          Virtual Address

각 프로세스는 독립적인 사용자 virtual page 집합을 가집니다. 사용자 페이지는 가상 주소 KERN_BASE(0x8004000000) 아래에 있는 페이지입니다. 반대로 커널 virtual page 집합은 전역이며, 어떤 스레드나 프로세스가 실행 중이든 같은 위치에 남아 있습니다. 커널은 사용자 페이지와 커널 페이지 모두에 접근할 수 있지만, 사용자 프로세스는 자기 사용자 페이지에만 접근할 수 있습니다. 자세한 내용은 가상 메모리 배치를 참고하세요.

Pintos는 가상 주소를 다루기 위한 여러 유용한 함수를 제공합니다. 자세한 내용은 부록의 가상 주소를 참고하세요.

프레임

frame, physical frame, 또는 page frame은 물리 메모리의 연속된 영역입니다. 페이지와 마찬가지로 프레임도 page size 크기여야 하고 page-aligned되어야 합니다. 따라서 64비트 물리 주소는 다음처럼 frame number와 frame offset, 또는 그냥 offset으로 나눌 수 있습니다.

                          12 11         0
    +-----------------------+-----------+
    |      Frame Number     |   Offset  |
    +-----------------------+-----------+
              Physical Address

x86-64는 물리 주소의 메모리에 직접 접근하는 방법을 제공하지 않습니다. Pintos는 커널 가상 메모리를 물리 메모리에 직접 매핑해서 이 문제를 해결합니다. 커널 가상 메모리의 첫 번째 페이지는 물리 메모리의 첫 번째 프레임에, 두 번째 페이지는 두 번째 프레임에 매핑되는 식입니다. 따라서 프레임은 커널 가상 메모리를 통해 접근할 수 있습니다.

Pintos는 물리 주소와 커널 가상 주소를 서로 변환하는 함수들을 제공합니다. 자세한 내용은 가상 주소를 참고하세요.

페이지 테이블

page table은 CPU가 가상 주소를 물리 주소로, 즉 page를 frame으로 변환할 때 사용하는 자료구조입니다. page table 형식은 x86-64 아키텍처가 정합니다. Pintos는 threads/mmu.c에 page table 관리 코드를 제공합니다.

아래 그림은 page와 frame의 관계를 보여 줍니다. 왼쪽의 가상 주소는 page number와 offset으로 이루어져 있습니다. page table은 page number를 frame number로 변환하고, 변하지 않은 offset과 합쳐서 오른쪽의 물리 주소를 얻습니다.

                          +----------+
         .--------------->|Page Table|-----------.
        /                 +----------+            |
        |   12 11 0                               V  12 11 0
    +---------+----+                         +---------+----+
    | Page Nr | Ofs|                         |Frame Nr | Ofs|
    +---------+----+                         +---------+----+
     Virt Addr   |                            Phys Addr    ^
                  \_______________________________________/

스왑 슬롯

swap slot은 swap partition 안의 page size 크기 디스크 공간입니다. 하드웨어 제약은 frame보다 slot 배치에 더 유연하지만, page-aligned로 두어도 손해가 없으므로 swap slot도 page-aligned로 두는 것이 좋습니다.

반드시 세 자료구조를 완전히 분리해서 구현할 필요는 없습니다. 관련 자원을 하나의 통합 자료구조로 전부 또는 일부 합치는 편이 편할 수도 있습니다.

각 자료구조에 대해, 각 원소가 어떤 정보를 담아야 하는지 결정해야 합니다. 또한 그 자료구조의 범위가 local인지, 즉 프로세스별인지, 아니면 global인지, 즉 시스템 전체에 적용되는지 결정해야 하고, 그 범위 안에서 인스턴스가 몇 개 필요한지도 정해야 합니다.

설계를 단순하게 하기 위해, 이런 자료구조들은 non-pageable memory에 저장해도 됩니다. 예를 들어 calloc이나 malloc으로 할당한 메모리를 사용할 수 있습니다. 그러면 자료구조들 사이의 포인터가 계속 유효하다고 확신할 수 있습니다.

구현 선택지: 성능 관점

구현에 사용할 수 있는 선택지에는 array, list, bitmap, hash table 등이 있습니다. array는 보통 가장 단순하지만, 듬성듬성 채워진 array는 메모리를 낭비합니다. list도 단순하지만, 긴 list에서 특정 위치를 찾으려면 순회 시간이 낭비됩니다. array와 list는 둘 다 크기를 조절할 수 있지만, list는 중간 삽입과 삭제를 더 효율적으로 지원합니다.

Pintos에는 lib/kernel/bitmap.c와 include/lib/kernel/bitmap.h에 bitmap 자료구조가 있습니다. bitmap은 bit의 배열이며, 각 bit는 true 또는 false가 될 수 있습니다. bitmap은 보통 동일한 자원 집합의 사용 상태를 추적하는 데 사용됩니다. 자원 n이 사용 중이면 bitmap의 bit n이 true입니다. Pintos bitmap은 고정 크기지만, 필요하면 resize를 지원하도록 확장할 수도 있습니다.

Pintos에는 hash table 자료구조도 있습니다. 해시 테이블을 참고하세요. Pintos hash table은 넓은 크기 범위에서 삽입과 삭제를 효율적으로 지원합니다.

더 복잡한 자료구조는 더 나은 성능이나 다른 이점을 줄 수 있지만, 구현을 불필요하게 복잡하게 만들 수도 있습니다. 따라서 balanced binary tree 같은 고급 자료구조를 설계 일부로 구현하는 것은 권장하지 않습니다.

Supplemental Page Table 관리

supplemental page table은 page table에 각 page에 대한 추가 정보를 보완합니다. page table 형식의 제약 때문에 이런 자료구조가 필요합니다. 이런 자료구조도 흔히 "page table"이라고 부르지만, 혼동을 줄이기 위해 여기서는 "supplemental"이라는 말을 붙입니다.

supplemental page table은 적어도 두 가지 목적으로 사용됩니다. 가장 중요한 목적은 page fault가 발생했을 때입니다. 커널은 fault가 난 virtual page를 supplemental page table에서 찾아 그 page에 어떤 데이터가 들어 있어야 하는지 알아냅니다. 두 번째로, 프로세스가 종료될 때 커널은 supplemental page table을 참고해서 어떤 자원을 해제해야 하는지 결정합니다.

Supplemental Page Table 구성

supplemental page table은 원하는 방식으로 구성할 수 있습니다. 기본 접근법은 적어도 두 가지입니다. segment 단위로 구성하는 방법과 page 단위로 구성하는 방법입니다. 여기서 segment는 연속된 page들의 묶음, 즉 실행 파일이나 memory-mapped file을 포함하는 메모리 영역을 뜻합니다.

선택적으로 page table 자체를 이용해서 supplemental page table의 구성원을 추적할 수도 있습니다. 이를 위해서는 threads/mmu.c의 Pintos page table 구현을 수정해야 합니다. 이 접근은 숙련된 학생에게만 권장합니다.

Page Fault 처리

supplemental page table의 가장 중요한 사용자는 page fault handler입니다. 프로젝트 2에서는 page fault가 항상 커널이나 사용자 프로그램의 버그를 의미했습니다. 프로젝트 3에서는 더 이상 그렇지 않습니다. 이제 page fault는 어떤 page를 파일이나 swap slot에서 가져와야 한다는 뜻일 수도 있습니다. 이런 경우를 처리하려면 더 정교한 page fault handler를 구현해야 합니다. userprog/exception.c의 page_fault()는 vm/vm.c의 vm_try_handle_fault(), 즉 여러분의 page fault handler를 호출합니다. 이 handler는 대략 다음 일을 해야 합니다.

  • fault가 난 page를 supplemental page table에서 찾습니다. 메모리 참조가 유효하다면 supplemental page table entry를 이용해 page에 들어갈 데이터를 찾습니다. 그 데이터는 파일 시스템에 있을 수도 있고, swap slot에 있을 수도 있고, 단순히 모두 0인 page일 수도 있습니다. sharing, 즉 Copy-on-Write를 구현한다면 해당 page의 데이터가 이미 page frame 안에 있지만 page table에는 없을 수도 있습니다. supplemental page table이 사용자 프로세스가 접근하려던 주소에 어떤 데이터도 기대하면 안 된다고 나타내거나, page가 커널 가상 메모리 안에 있거나, read-only page에 쓰려는 접근이라면 그 접근은 유효하지 않습니다. 유효하지 않은 접근은 프로세스를 종료하고 그 모든 자원을 해제합니다.
  • page를 저장할 frame을 얻습니다. sharing을 구현했다면 필요한 데이터가 이미 어떤 frame에 있을 수 있으므로, 그 frame을 찾을 수 있어야 합니다.
  • 파일 시스템이나 swap에서 읽어 오거나, 0으로 채우는 등의 방식으로 데이터를 frame에 가져옵니다. sharing을 구현했다면 필요한 page가 이미 frame 안에 있을 수 있으므로 이 단계에서 아무 작업도 필요하지 않을 수 있습니다.
  • fault가 난 가상 주소의 page table entry가 해당 physical page를 가리키게 합니다. threads/mmu.c의 함수들을 사용할 수 있습니다.

Frame Table 관리

frame table은 각 frame마다 하나의 entry를 가집니다. 각 entry에는 현재 그 frame을 차지하고 있는 page가 있다면 그 page를 가리키는 포인터와, 설계에 따라 필요한 다른 데이터가 들어갑니다. frame table은 빈 frame이 없을 때 내보낼 page를 선택해서 Pintos가 eviction policy를 효율적으로 구현할 수 있게 해 줍니다.

사용자 page에 쓰는 frame은 palloc_get_page(PAL_USER)를 호출해서 "user pool"에서 얻어야 합니다. 반드시 PAL_USER를 사용해야 합니다. "kernel pool"에서 할당하면 일부 테스트 케이스가 예상치 못하게 실패할 수 있습니다. frame table 구현의 일부로 palloc.c를 수정한다면, 두 pool의 구분은 반드시 유지하세요.

frame table의 가장 중요한 연산은 사용하지 않는 frame을 얻는 것입니다. 빈 frame이 있으면 쉽습니다. 빈 frame이 하나도 없으면 어떤 page를 frame에서 evict해서 frame을 비워야 합니다.

swap slot을 할당하지 않고는 evict할 수 있는 frame이 없는데 swap도 가득 찬 경우, 커널을 panic시킵니다. 실제 OS는 이런 상황을 복구하거나 예방하기 위해 다양한 정책을 사용하지만, 이 프로젝트 범위를 벗어납니다.

eviction 과정은 대략 다음 단계로 이루어집니다.

  • page replacement algorithm을 사용해 evict할 frame을 선택합니다. 아래에서 설명하는 page table의 "accessed" bit와 "dirty" bit가 유용합니다.
  • 그 frame을 참조하는 모든 page table에서 frame에 대한 참조를 제거합니다. sharing을 구현하지 않았다면 어느 시점이든 하나의 page만 하나의 frame을 참조해야 합니다.
  • 필요하다면 page를 파일 시스템이나 swap에 씁니다. 그 뒤 evict된 frame은 다른 page를 저장하는 데 사용할 수 있습니다.

Accessed Bit와 Dirty Bit

x86-64 하드웨어는 각 page의 page table entry(PTE)에 있는 두 bit를 통해 page replacement algorithm 구현을 어느 정도 도와줍니다. 어떤 page를 읽거나 쓰면 CPU는 그 page의 PTE에서 accessed bit를 1로 설정합니다. 어떤 page에 쓰면 dirty bit를 1로 설정합니다. CPU는 이 bit들을 절대 0으로 되돌리지 않지만, OS는 그렇게 할 수 있습니다.

alias에도 주의해야 합니다. alias란 두 개 이상의 page가 같은 frame을 참조하는 경우입니다. alias된 frame에 접근하면 accessed bit와 dirty bit는 접근에 사용된 page의 page table entry 하나에서만 갱신됩니다. 다른 alias의 accessed bit와 dirty bit는 갱신되지 않습니다.

Pintos에서는 모든 사용자 virtual page가 자신의 kernel virtual page와 alias되어 있습니다. 이 alias를 어떤 방식으로든 관리해야 합니다. 예를 들어 두 주소 모두의 accessed bit와 dirty bit를 확인하고 갱신할 수 있습니다. 또는 커널이 사용자 데이터를 항상 사용자 가상 주소를 통해서만 접근하게 해서 문제를 피할 수도 있습니다.

그 밖의 alias는 sharing을 구현했거나 코드에 버그가 있을 때만 생겨야 합니다.

accessed bit와 dirty bit를 다루는 함수에 대한 자세한 내용은 부록의 page table accessed/dirty bit 설명을 참고하세요.

Swap Table 관리

swap table은 사용 중인 swap slot과 비어 있는 swap slot을 추적합니다. frame에서 page를 evict해 swap partition으로 보낼 때 사용하지 않는 swap slot을 고를 수 있어야 합니다. page를 다시 읽어 왔거나, swap된 page를 가진 프로세스가 종료되었을 때는 swap slot을 해제할 수 있어야 합니다.

vm/build 디렉터리에서 pintos-mkdisk swap.dsk --swap-size=n 명령을 사용하면 n MB 크기의 swap partition을 담은 swap.dsk 디스크를 만들 수 있습니다. 그 뒤 pintos를 실행하면 swap.dsk가 자동으로 추가 디스크로 붙습니다. 또는 --swap-size=n 옵션으로 한 번 실행하는 동안만 사용할 임시 n MB swap disk를 지정할 수도 있습니다.

swap slot은 lazy하게 할당해야 합니다. 즉 eviction 때문에 실제로 필요할 때만 할당해야 합니다. 프로세스 시작 시점에 실행 파일에서 data page를 읽고 곧바로 swap에 쓰는 것은 lazy하지 않습니다. 특정 page를 저장하기 위해 swap slot을 미리 예약해서도 안 됩니다.

swap slot의 내용이 frame으로 다시 읽혀 들어오면 그 swap slot을 해제하세요.

Memory Mapped File 관리

파일 시스템에 접근하는 가장 일반적인 방법은 read와 write 시스템 콜입니다. 보조 인터페이스로 mmap 시스템 콜을 사용해 파일을 virtual page에 "map"할 수 있습니다. 그러면 프로그램은 파일 데이터에 대해 메모리 명령을 직접 사용할 수 있습니다. 예를 들어 파일 foo의 길이가 0x1000바이트, 즉 4 kB 한 페이지라고 합시다. foo가 주소 0x5000부터 메모리에 map되면, 0x5000부터 0x5fff까지의 위치에 대한 메모리 접근은 foo의 해당 바이트에 접근합니다.

아래 프로그램은 mmap을 사용해 파일을 콘솔에 출력합니다. 명령줄에서 지정한 파일을 열고, 가상 주소 0x10000000에 map한 뒤, map된 데이터를 콘솔(fd 1)에 쓰고, 파일 mapping을 해제합니다.

#include <stdio.h>
#include <syscall.h>
int main (int argc UNUSED, char *argv[])
{
  void *data = (void *) 0x10000000;                 /* Address at which to map. */
  int fd = open (argv[1]);                          /* Open file. */
  void *map = mmap (data, filesize (fd), 0, fd, 0); /* Map file. */
  write (1, data, filesize (fd));                   /* Write file to console. */
  munmap (map);                                     /* Unmap file (optional). */
  return 0;
}

제출물은 memory mapped file이 사용하는 메모리를 추적할 수 있어야 합니다. 그래야 mapped region에서 page fault를 올바르게 처리하고, mapped file이 프로세스 안의 다른 segment와 겹치지 않도록 보장할 수 있습니다.
