Memory Mapped Files

이 절에서는 memory-mapped page를 구현합니다. anonymous page와 달리 memory-mapped page는 file-backed mapping입니다. page의 내용은 기존 파일의 데이터를 반영합니다. page fault가 발생하면 physical frame이 즉시 할당되고, 파일의 내용이 메모리로 복사됩니다. memory-mapped page가 unmap되거나 swap out되면, 내용의 변경 사항이 파일에 반영됩니다.

mmap과 munmap 시스템 콜

memory mapped file을 위한 두 시스템 콜인 mmap과 munmap을 구현하세요. VM 시스템은 mmap 영역의 page들을 lazy하게 로드해야 하며, mapping의 backing store로 mmap된 파일 자체를 사용해야 합니다. 이 두 시스템 콜을 구현할 때는 vm/file.c에 정의된 do_mmap과 do_munmap을 구현하고 사용해야 합니다.

void *mmap (void *addr, size_t length, int writable, int fd, off_t offset);

fd로 열린 파일에서 offset 바이트부터 시작하는 length 바이트를, 프로세스의 가상 주소 공간에서 addr에 mapping합니다. 전체 파일은 addr에서 시작하는 연속된 virtual page들에 mapping됩니다. 파일 길이가 PGSIZE의 배수가 아니면, 마지막 mapped page의 일부 바이트가 파일 끝을 넘어 "튀어나오게" 됩니다. 해당 page가 fault로 메모리에 올라올 때 이 바이트들은 0으로 설정하고, page를 디스크에 다시 쓸 때는 버립니다. 성공하면 이 함수는 파일이 mapping된 가상 주소를 반환합니다. 실패하면 파일 mapping에 유효한 주소가 아닌 NULL을 반환해야 합니다.

fd로 열린 파일의 길이가 0바이트이면 mmap 호출은 실패할 수 있습니다. addr이 page-aligned되어 있지 않거나, mapping하려는 page 범위가 이미 mapping된 page 집합과 겹치면 반드시 실패해야 합니다. 여기에는 stack과 실행 파일 로드 시점에 mapping된 page들도 포함됩니다. Linux에서는 addr이 NULL이면 커널이 mapping을 만들 적절한 주소를 찾습니다. 단순화를 위해 Pintos에서는 주어진 addr에서만 mmap을 시도하면 됩니다. 따라서 addr이 0이면 반드시 실패해야 합니다. Pintos의 일부 코드는 virtual page 0이 mapping되지 않았다고 가정하기 때문입니다. length가 0일 때도 mmap은 실패해야 합니다. 마지막으로 콘솔 입력과 출력을 나타내는 파일 디스크립터는 mapping할 수 없습니다.

memory-mapped page도 anonymous page처럼 lazy하게 할당되어야 합니다. page object를 만들 때 vm_alloc_page_with_initializer나 vm_alloc_page를 사용할 수 있습니다.

void munmap (void *addr);

지정된 주소 범위 addr의 mapping을 해제합니다. addr은 같은 프로세스가 이전에 mmap을 호출해서 받은 가상 주소여야 하며, 아직 unmap되지 않은 mapping이어야 합니다.

프로세스가 exit로 종료되든 다른 방법으로 종료되든, 모든 mapping은 프로세스 종료 시 암묵적으로 unmap됩니다. mapping이 명시적으로든 암묵적으로든 unmap될 때, 프로세스가 쓴 모든 page는 파일에 다시 기록되어야 하고, 쓰지 않은 page는 기록되면 안 됩니다. 그런 다음 해당 page들은 프로세스의 virtual page 목록에서 제거됩니다.

파일을 닫거나 삭제해도 그 파일의 mapping은 unmap되지 않습니다. Unix 관례에 따라, 한 번 만들어진 mapping은 munmap이 호출되거나 프로세스가 종료될 때까지 유효합니다. 자세한 내용은 열린 파일 삭제를 참고하세요. 각 mapping마다 파일에 대한 별도이고 독립적인 참조를 얻기 위해 file_reopen 함수를 사용해야 합니다.

두 개 이상의 프로세스가 같은 파일을 mapping하더라도, 그들이 일관된 데이터를 봐야 한다는 요구사항은 없습니다. Unix는 두 mapping이 같은 physical page를 공유하게 해서 이를 처리하며, mmap 시스템 콜에는 page가 shared인지 private인지, 즉 copy-on-write인지 지정할 수 있는 인자도 있습니다.

필요에 따라 vm/vm.c의 vm_file_init과 vm_file_initializer를 수정할 수 있습니다.

void vm_file_init (void);

file-backed page subsystem을 초기화합니다. 이 함수 안에서 file-backed page와 관련된 어떤 설정이든 할 수 있습니다.

bool file_backed_initializer (struct page *page, enum vm_type type, void *kva);

file-backed page를 초기화합니다. 이 함수는 먼저 page->operations에 file-backed page용 handler들을 설정합니다. 메모리를 backing하는 파일 같은 page 구조체의 정보를 갱신해야 할 수 있습니다.

static void file_backed_destroy (struct page *page);

연관된 파일을 닫아서 file-backed page를 파괴합니다. 내용이 dirty라면 변경 사항을 파일에 다시 기록해야 합니다. 이 함수 안에서 page 구조체 자체를 해제할 필요는 없습니다. file_backed_destroy의 호출자가 처리해야 합니다.
