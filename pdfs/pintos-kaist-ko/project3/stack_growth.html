스택 증가

프로젝트 2에서 스택은 USER_STACK에서 시작하는 단일 page였고, 프로그램 실행은 이 크기로 제한되었습니다. 이제 스택이 현재 크기보다 더 자라면 필요한 만큼 추가 page를 할당합니다.

추가 page는 접근이 스택 접근으로 "보일" 때에만 할당하세요. 스택 접근과 다른 접근을 구별하려는 heuristic을 설계해야 합니다.

사용자 프로그램이 stack pointer 아래의 스택에 쓰는 것은 버그입니다. 일반적인 실제 OS는 언제든지 프로세스에 "signal"을 전달하기 위해 프로세스를 interrupt할 수 있고, signal 전달은 스택의 데이터를 수정할 수 있기 때문입니다. 하지만 x86-64 PUSH 명령은 stack pointer를 조정하기 전에 접근 권한을 확인하므로, stack pointer보다 8바이트 아래에서 page fault를 일으킬 수 있습니다.

사용자 프로그램의 현재 stack pointer 값을 얻을 수 있어야 합니다. 사용자 프로그램이 만든 시스템 콜 안이나 사용자 프로그램이 일으킨 page fault 안에서는, 각각 syscall_handler() 또는 page_fault()에 전달된 struct intr_frame의 rsp 멤버에서 값을 가져올 수 있습니다. 잘못된 메모리 접근을 감지하는 데 page fault에 의존한다면, page fault가 커널 안에서 발생하는 경우도 처리해야 합니다. 프로세서는 예외 때문에 user mode에서 kernel mode로 전환될 때에만 stack pointer를 저장합니다. 따라서 커널 안에서 발생한 page fault에 대해 page_fault()에 전달된 struct intr_frame의 rsp를 읽으면 사용자 stack pointer가 아니라 정의되지 않은 값을 얻게 됩니다. user mode에서 kernel mode로 처음 전환될 때 rsp를 struct thread에 저장하는 식의 다른 방법을 마련해야 합니다.

스택 증가 기능을 구현합니다. 이를 위해 먼저 vm/vm.c의 vm_try_handle_fault를 수정해서 스택 증가 상황을 식별하세요. 스택 증가로 처리할 수 있다고 판단되면, 스택을 늘리기 위해 vm/vm.c의 vm_stack_growth를 호출해야 합니다. vm_stack_growth를 구현하세요.

bool vm_try_handle_fault (struct intr_frame *f, void *addr,
    bool user, bool write, bool not_present);

이 함수는 userprog/exception.c의 page_fault가 page fault 예외를 처리하는 중에 호출됩니다. 이 함수 안에서는 page fault가 스택 증가로 처리할 수 있는 유효한 경우인지 확인해야 합니다. fault를 스택 증가로 처리할 수 있다고 확인했다면, fault가 발생한 주소로 vm_stack_growth를 호출하세요.

void vm_stack_growth (void *addr);

addr이 더 이상 fault 주소가 아니도록 하나 이상의 anonymous page를 할당해서 스택 크기를 늘립니다. 할당을 처리할 때는 addr를 PGSIZE 단위로 내림 정렬해야 합니다.

대부분의 OS는 스택 크기에 절대적인 제한을 둡니다. 일부 OS에서는 많은 Unix 시스템의 ulimit 명령처럼 사용자가 이 제한을 조정할 수 있게 합니다. 많은 GNU/Linux 시스템의 기본 제한은 8 MB입니다. 이 프로젝트에서는 스택 크기를 최대 1 MB로 제한해야 합니다.

이제 모든 stack-growth 테스트 케이스가 통과해야 합니다.
