KAIST Pintos GitBook 원문을 한국어로 옮긴 로컬 읽기본입니다. 코드, 명령어, 함수명, 파일명은 의도적으로 원문 형태를 최대한 유지했습니다. 의미가 애매한 부분은 원문 링크와 함께 확인하세요.

채점

과제는 테스트 결과를 기준으로 채점합니다.

테스트

테스트 점수는 강사진이 제공하는 테스트를 기준으로 결정됩니다. 각 프로젝트에는 여러 테스트가 있고, 각 테스트 이름은 tests로 시작합니다.

제출물을 전체적으로 테스트하려면 해당 프로젝트의 build 디렉터리에서 make check를 실행합니다. 이 명령은 각 테스트를 빌드하고 실행한 뒤, 테스트마다 pass 또는 fail 메시지를 출력합니다. 테스트가 실패하면 실패 이유에 대한 세부 정보도 함께 출력합니다. 모든 테스트가 끝나면 결과 요약도 출력됩니다.

개별 테스트만 따로 실행할 수도 있습니다. 어떤 테스트를 t라고 하면, 테스트 출력은 t.output에 저장되고, 채점 스크립트가 그 출력을 보고 pass 또는 fail 판정을 t.result에 씁니다. 단일 테스트를 실행하고 채점하려면 build 디렉터리에서 해당 .result 파일을 명시적으로 만들면 됩니다. 예를 들어 make tests/threads/alarm-multiple.result처럼 실행합니다.

make가 테스트 결과가 이미 최신이라고 말하지만 다시 실행하고 싶다면, make clean을 실행하거나 해당 .output 파일을 직접 삭제하세요. 기본적으로 각 테스트는 실행 도중이 아니라 완료 시점에만 피드백을 줍니다. 실행 과정을 보고 싶다면 make check VERBOSE=1처럼 make 명령줄에 VERBOSE=1을 지정하면 됩니다.

모든 테스트와 관련 파일은 pintos/src/tests에 있습니다. 강사진은 제출물을 테스트하기 전에 이 디렉터리의 내용을 깨끗하고 수정되지 않은 원본 테스트로 교체합니다. 따라서 디버깅을 위해 일부 테스트를 수정해 볼 수는 있지만, 실제 채점에는 원본 테스트가 사용됩니다.

모든 소프트웨어에는 버그가 있으므로 강사진의 테스트에도 결함이 있을 수 있습니다. 테스트 실패가 여러분 코드의 버그가 아니라 테스트 자체의 버그라고 생각한다면 알려 주세요. 확인한 뒤 필요하면 수정합니다.

다만 테스트 스위트를 공개해 주는 것을 악용해서는 안 됩니다. 코드는 제공된 테스트 케이스만 통과하는 것이 아니라 일반적인 경우에도 올바르게 동작해야 합니다. 예를 들어 실행 중인 테스트 케이스 이름을 보고 커널 동작을 바꾸는 방식은 허용되지 않습니다. 이런 식으로 테스트를 우회하려는 시도는 점수를 받을 수 없습니다. 본인의 풀이가 애매한 영역에 있다고 생각되면 강사진에게 질문하세요.

제출

아카이브 파일을 만드는 명령이 제공됩니다. threads, userprog, vm, filesys 디렉터리가 있는 Pintos 프로젝트 루트 디렉터리로 이동하세요. 그곳에서 TEAM=YOUR_TEAM_NUMBER make archive를 실행합니다. 예를 들어 팀 번호가 5라면 TEAM=5 make archive를 실행합니다. 그러면 같은 디렉터리에 team5.tar.gz가 생성됩니다. 이 아카이브 파일을 제출하면 됩니다.

중복 제출을 하지 마세요. 한 팀당 하나의 제출이면 충분합니다.
