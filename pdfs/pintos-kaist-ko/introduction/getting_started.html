시작하기

시작하려면 Pintos를 빌드할 수 있는 머신에 로그인해야 합니다. 강사진이 제공한 Linux 머신을 사용해도 되고, 본인의 Solaris 또는 Linux 머신을 사용해도 됩니다. 다만 채점은 제공된 머신에서 수행되며, 이 문서의 안내도 그 환경을 기준으로 합니다. 개인 머신에 Pintos를 설치하거나 그 위에서 작업하는 과정은 지원 대상이 아닙니다. 아래와 같은 서버 환경이 각 학생에게 제공됩니다.

다음 명령을 실행하면 Pintos 소스 코드를 pintos라는 디렉터리에 받을 수 있습니다.

git clone https://github.com/casys-kaist/pintos-kaist

프로젝트 기간이 시작되기 전에 소스 코드가 바뀔 가능성이 있으므로, 저장소를 복제해 두는 것을 권장합니다. 템플릿 코드가 바뀌면 본인 코드와 템플릿을 쉽게 동기화할 수 있기 때문입니다. 단, 공개 포크를 만들면 안 됩니다. 프로젝트 기간 중에는 템플릿 코드에 치명적인 버그가 있는 경우를 제외하고 소스 코드를 변경하지 않습니다.

참고로 Pintos 코드와 풀이 코드는 제공된 Linux 머신과 같은 환경인 Ubuntu 16.04.6 LTS, gcc (Ubuntu 7.4.0-1ubuntu1~16.04~ppa1) 7.4.0, QEMU emulator version 2.5.0 (Debian 1:2.5+dfsg-5ubuntu10.43)에서 테스트되었습니다.

소스 코드를 받은 뒤에는 Pintos 루트 디렉터리에서 아래 명령으로 환경을 설정할 수 있습니다.

$ source ./activate

이 줄을 홈 디렉터리의 .bashrc에 추가해 두는 것이 좋습니다. 그렇지 않으면 로그인할 때마다 직접 입력해야 합니다.

소스 트리 개요

이제 내부 구조를 살펴봅니다. pintos/ 디렉터리 안에는 대략 다음 구조가 있습니다.

  • threads/: 기본 커널의 소스 코드입니다. Project 1부터 이 코드를 수정합니다.
  • userprog/: 사용자 프로그램 로더의 소스 코드입니다. Project 2부터 이 코드를 수정합니다.
  • vm/: 거의 비어 있는 디렉터리입니다. Project 3에서 이곳에 가상 메모리를 구현합니다.
  • filesys/: 기본 파일 시스템의 소스 코드입니다. Project 2부터 이 파일 시스템을 사용하지만, Project 4 전까지는 수정하지 않습니다.
  • devices/: 키보드, 타이머, 디스크 같은 I/O 장치 인터페이스 코드입니다. Project 1에서 타이머 구현을 수정합니다. 그 외에는 보통 이 코드를 바꿀 필요가 없습니다.
  • lib/: 표준 C 라이브러리 일부를 구현한 코드입니다. 이 디렉터리의 코드는 Pintos 커널에도 컴파일되고, Project 2부터는 Pintos 위에서 실행되는 사용자 프로그램에도 컴파일됩니다. 보통 이 코드를 수정할 일은 거의 없습니다.
  • include/lib/kernel/: Pintos 커널에서만 사용하는 C 라이브러리 부분입니다. 비트맵, 이중 연결 리스트, 해시 테이블처럼 커널 코드에서 자유롭게 사용할 수 있는 자료구조 구현도 포함합니다. 커널에서는 이 디렉터리의 헤더를 #include <...> 형태로 포함할 수 있습니다.
  • include/lib/user/: Pintos 사용자 프로그램에서만 사용하는 C 라이브러리 부분입니다. 사용자 프로그램에서는 이 디렉터리의 헤더를 #include <...> 형태로 포함할 수 있습니다.
  • tests/: 각 프로젝트의 테스트입니다. 제출물을 테스트하는 데 도움이 된다면 이 코드를 수정해 볼 수 있습니다. 다만 실제 채점 전에는 원본 테스트로 교체됩니다.
  • examples/: Project 2부터 사용할 수 있는 예제 사용자 프로그램입니다.
  • include/: 헤더 파일(*.h)의 소스 코드입니다.

Pintos 빌드하기

다음 단계는 첫 번째 프로젝트에 제공된 소스 코드를 빌드하는 것입니다. 먼저 threads 디렉터리로 이동한 뒤 make 명령을 실행합니다. 그러면 threads 아래에 build 디렉터리가 만들어지고, 그 안에 Makefile과 몇몇 하위 디렉터리가 채워진 뒤 커널이 빌드됩니다.

빌드가 끝나면 build 디렉터리에서 다음 파일들이 중요합니다.

  • Makefile: pintos/src/Makefile.build의 복사본입니다. 커널을 어떻게 빌드하는지 설명합니다.
  • kernel.o: 전체 커널의 오브젝트 파일입니다. 각 커널 소스 파일에서 컴파일된 오브젝트 파일들을 하나의 오브젝트 파일로 링크한 결과입니다. 디버그 정보를 포함하므로, 이 파일에 대해 GDB(GDB 참조)나 backtrace(Backtraces 참조)를 실행할 수 있습니다.
  • kernel.bin: 커널의 메모리 이미지입니다. 즉 Pintos 커널을 실행하기 위해 메모리에 실제로 로드되는 바이트들입니다. 기본적으로 kernel.o에서 디버그 정보를 제거한 파일입니다. 이렇게 하면 공간을 많이 절약할 수 있고, 커널 로더 설계상 부과되는 512 kB 크기 제한에 걸릴 가능성을 줄입니다.
  • loader.bin: 커널 로더의 메모리 이미지입니다. 디스크에서 커널을 읽어 메모리에 올리고 실행을 시작하는 작은 어셈블리 코드입니다. 길이는 정확히 512바이트이며, 이 크기는 PC BIOS에 의해 정해집니다. build의 하위 디렉터리에는 컴파일러가 만든 오브젝트 파일(.o)과 의존성 파일(.d)이 들어 있습니다. 의존성 파일은 다른 소스 파일이나 헤더 파일이 바뀌었을 때 어떤 소스 파일을 다시 컴파일해야 하는지 make에게 알려 줍니다.

Pintos 실행하기

시뮬레이터에서 Pintos를 편하게 실행할 수 있도록 pintos라는 프로그램이 제공됩니다. 가장 단순한 형태로는 pintos argument...처럼 실행할 수 있습니다. 각 인자는 Pintos 커널에 전달되어 커널이 처리합니다.

직접 실행해 봅시다. 먼저 새로 만들어진 build 디렉터리로 이동합니다. 그 다음 pintos run alarm-multiple 명령을 실행합니다. 이 명령은 run alarm-multiple이라는 인자를 Pintos 커널에 전달합니다. 여기서 run은 커널에게 테스트를 실행하라고 지시하고, alarm-multiple은 실행할 테스트 이름입니다. Pintos가 부팅된 뒤 alarm-multiple 테스트 프로그램을 실행하고, 몇 화면 분량의 텍스트를 출력합니다.

명령줄 리디렉션을 사용하면 serial 출력을 파일로 저장할 수 있습니다. 예를 들어 pintos -- run alarm-multiple > logfile처럼 실행합니다. pintos 프로그램은 QEMU나 가상 하드웨어를 설정하기 위한 여러 옵션도 제공합니다. 옵션을 지정한다면 Pintos 커널에 전달할 명령보다 앞에 와야 하며, 둘은 --로 구분해야 합니다. 전체 형태는 pintos option... -- argument...입니다.

사용 가능한 옵션 목록을 보려면 인자 없이 pintos를 실행하세요. 예를 들어 VM 출력을 표시하는 방식을 바꾸는 옵션이 있습니다. -v는 VGA 디스플레이를 끄고, -s는 표준 입력에서 serial 입력을 받거나 표준 출력으로 serial 출력을 내보내는 동작을 억제합니다. Pintos 커널에는 run 외에도 다른 명령과 옵션이 있습니다. 지금은 중요하지 않지만, pintos -h처럼 -h를 사용하면 목록을 볼 수 있습니다.
