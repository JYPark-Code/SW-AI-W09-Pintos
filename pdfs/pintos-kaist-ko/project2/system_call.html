원문 보기
KAIST Pintos GitBook 원문을 한국어로 옮긴 로컬 읽기본입니다. 코드, 명령어, 함수명, 파일명은 의도적으로 원문 형태를 최대한 유지했습니다. 의미가 애매한 부분은 원문 링크와 함께 확인하세요.

시스템 콜

시스템 콜 기반 구조를 구현합니다.

userprog/syscall.c에 시스템 콜 핸들러를 구현하세요. 제공된 스켈레톤 구현은 시스템 콜을 "처리"한다는 명목으로 프로세스를 종료하기만 합니다. 실제 구현에서는 시스템 콜 번호를 가져오고, 필요한 시스템 콜 인자들을 가져온 뒤, 그에 맞는 동작을 수행해야 합니다.

시스템 콜 세부 사항

프로젝트 1에서는 운영체제가 사용자 프로그램으로부터 제어권을 되찾는 한 가지 방법을 이미 다뤘습니다. 타이머와 I/O 장치에서 오는 인터럽트입니다. 이런 인터럽트는 CPU 바깥의 대상이 일으키므로 "external" interrupt입니다.

운영체제는 소프트웨어 예외도 처리합니다. 소프트웨어 예외는 프로그램 코드 실행 중 발생하는 이벤트입니다. page fault나 0으로 나누기 같은 오류가 여기에 해당합니다. 예외는 사용자 프로그램이 운영체제에 서비스, 즉 "시스템 콜"을 요청하는 수단이기도 합니다.

전통적인 x86 아키텍처에서는 시스템 콜도 다른 소프트웨어 예외와 같은 방식으로 처리했습니다. 하지만 x86-64에서는 제조사들이 시스템 콜을 위한 특수 명령어인 syscall을 도입했습니다. 이 명령어는 시스템 콜 핸들러를 빠르게 호출하는 방법을 제공합니다.

오늘날 x86-64에서 시스템 콜을 호출하는 가장 일반적인 방법은 syscall 명령어입니다. Pintos에서도 사용자 프로그램은 시스템 콜을 수행하기 위해 syscall을 호출합니다. syscall 명령을 실행하기 전, 시스템 콜 번호와 추가 인자는 일반 호출 규약과 비슷하게 레지스터에 설정되어 있어야 합니다. 다만 두 가지 차이가 있습니다.

  • %rax가 시스템 콜 번호입니다.
  • 네 번째 인자는 %rcx가 아니라 %r10입니다.

따라서 시스템 콜 핸들러 syscall_handler()가 제어권을 얻었을 때, 시스템 콜 번호는 rax에 있고 인자들은 %rdi, %rsi, %rdx, %r10, %r8, %r9 순서로 전달됩니다.

호출자의 레지스터 값은 핸들러에 전달되는 struct intr_frame을 통해 접근할 수 있습니다. struct intr_frame은 커널 스택 위에 있습니다.

x86-64의 함수 반환값 규약은 반환값을 RAX 레지스터에 두는 것입니다. 값을 반환하는 시스템 콜은 struct intr_frame의 rax 멤버를 수정해서 반환값을 전달할 수 있습니다.

다음 시스템 콜을 구현하세요

아래 프로토타입은 사용자 프로그램이 include/lib/user/syscall.h를 포함했을 때 보게 되는 형태입니다. 이 헤더와 include/lib/user 안의 다른 헤더들은 사용자 프로그램 전용입니다. 각 시스템 콜의 번호는 include/lib/syscall-nr.h에 정의되어 있습니다.

void halt (void);

src/include/threads/init.h에 선언된 power_off()를 호출해서 Pintos를 종료합니다. 이 호출은 가능한 한 드물게 사용해야 합니다. deadlock 같은 상황을 분석할 수 있는 정보를 잃게 될 수 있기 때문입니다.

void exit (int status);

현재 사용자 프로그램을 종료하고 status를 커널에 반환합니다. 부모 프로세스가 이 프로세스를 기다리면 아래 설명처럼 이 상태 값이 부모에게 반환됩니다. 관례적으로 상태 0은 성공을, 0이 아닌 값은 오류를 뜻합니다.

pid_t fork (const char *thread_name);

현재 프로세스를 복제한 새 프로세스를 만들고, 그 이름을 THREAD_NAME으로 지정합니다. 레지스터 값은 callee-saved register인 %RBX, %RSP, %RBP, %R12부터 %R15까지만 복제하면 됩니다.

부모 프로세스에서는 자식 프로세스의 pid를 반환해야 합니다. 실패하면 유효한 pid가 아닌 값을 반환해야 합니다. 자식 프로세스에서는 반환값이 0이어야 합니다. 자식은 파일 디스크립터와 가상 메모리 공간을 포함한 자원을 복제해서 가져야 합니다. 부모 프로세스는 자식 프로세스가 성공적으로 복제되었는지 알기 전까지 fork에서 반환하면 안 됩니다. 즉 자식 프로세스가 자원 복제에 실패하면 부모의 fork() 호출은 TID_ERROR를 반환해야 합니다.

템플릿은 threads/mmu.c의 pml4_for_each()를 사용해서 대응되는 page table 구조를 포함한 전체 사용자 메모리 공간을 복사합니다. 하지만 전달되는 pte_for_each_func의 빈 부분은 직접 채워야 합니다. 가상 주소를 참고하세요.

int exec (const char *cmd_line);

현재 프로세스를 cmd_line에 주어진 이름의 실행 파일로 바꾸고, 주어진 인자들도 함께 전달합니다. 성공하면 이 함수는 절대 반환하지 않습니다. 어떤 이유로든 프로그램을 로드하거나 실행할 수 없으면 프로세스는 종료 상태 -1로 종료됩니다. 이 함수는 exec를 호출한 스레드의 이름을 바꾸지 않습니다. 파일 디스크립터는 exec 호출 이후에도 열린 상태로 유지된다는 점에 주의하세요.

int wait (pid_t pid);

자식 프로세스 pid를 기다리고, 그 자식의 종료 상태를 가져옵니다. pid가 아직 살아 있으면 종료될 때까지 기다립니다. 그런 다음 pid가 exit에 전달한 상태 값을 반환합니다. pid가 exit()를 호출하지 않고 커널에 의해 종료되었다면, 예를 들어 예외 때문에 죽었다면, wait(pid)는 -1을 반환해야 합니다. 부모가 wait를 호출하기 전에 이미 종료된 자식 프로세스를 기다리는 것은 완전히 합법입니다. 그래도 커널은 부모가 그 자식의 종료 상태를 가져오거나, 자식이 커널에 의해 종료되었음을 알 수 있게 해야 합니다.

다음 조건 중 하나라도 참이면 wait는 즉시 실패하고 -1을 반환해야 합니다.

  • pid가 호출 프로세스의 직접 자식을 가리키지 않습니다. 호출 프로세스가 성공한 fork 호출의 반환값으로 pid를 받았을 때, 그리고 그때에만 pid는 호출 프로세스의 직접 자식입니다. 자식은 상속되지 않는다는 점에 주의하세요. A가 자식 B를 만들고 B가 자식 프로세스 C를 만들었다면, B가 죽었더라도 A는 C를 기다릴 수 없습니다. 프로세스 A의 wait(C) 호출은 실패해야 합니다. 마찬가지로 부모 프로세스가 먼저 종료되어 고아가 된 프로세스도 새 부모에게 배정되지 않습니다.
  • wait를 호출한 프로세스가 이미 같은 pid에 대해 wait를 호출했습니다. 즉 한 프로세스는 특정 자식에 대해 최대 한 번만 기다릴 수 있습니다.

프로세스는 원하는 수만큼 자식을 만들 수 있고, 어떤 순서로든 기다릴 수 있으며, 일부 또는 모든 자식을 기다리지 않고 종료할 수도 있습니다. 설계할 때는 wait가 일어날 수 있는 모든 경우를 고려해야 합니다. 부모가 기다렸는지 여부와 상관없이, 또 자식이 부모보다 먼저 종료되었는지 나중에 종료되었는지와 상관없이, 프로세스의 모든 자원은 struct thread를 포함해서 반드시 해제되어야 합니다.

초기 프로세스가 종료되기 전에는 Pintos가 종료되지 않도록 보장해야 합니다. 제공된 Pintos 코드는 threads/init.c의 main()에서 userprog/process.c의 process_wait()를 호출해 이를 처리하려고 합니다. 먼저 함수 위 주석에 맞게 process_wait()를 구현하고, 그 다음 wait 시스템 콜을 process_wait()를 이용해서 구현하는 것을 권장합니다.

이 시스템 콜 구현은 나머지 시스템 콜보다 훨씬 많은 작업을 요구합니다.

bool create (const char *file, unsigned initial_size);

file이라는 새 파일을 만들고 초기 크기를 initial_size 바이트로 설정합니다. 성공하면 true, 실패하면 false를 반환합니다. 새 파일을 만든다고 해서 그 파일이 열리는 것은 아닙니다. 새 파일을 여는 것은 별도의 open 시스템 콜이 필요한 작업입니다.

bool remove (const char *file);

file이라는 파일을 삭제합니다. 성공하면 true, 실패하면 false를 반환합니다. 파일이 열려 있든 닫혀 있든 삭제할 수 있으며, 열린 파일을 삭제해도 그 파일이 닫히지는 않습니다. 자세한 내용은 FAQ의 열린 파일 삭제(Removing an Open File)를 참고하세요.

int open (const char *file);

file이라는 파일을 엽니다. 성공하면 "file descriptor" 또는 fd라고 부르는 0 이상의 정수 핸들을 반환하고, 열 수 없으면 -1을 반환합니다. 파일 디스크립터 0과 1은 콘솔용으로 예약되어 있습니다. fd 0(STDIN_FILENO)은 표준 입력이고, fd 1(STDOUT_FILENO)은 표준 출력입니다. open 시스템 콜은 이 둘 중 어떤 값도 반환하지 않습니다. 이 값들은 아래에서 명시적으로 설명하는 경우에만 시스템 콜 인자로 유효합니다.

각 프로세스는 독립적인 파일 디스크립터 집합을 가집니다. 파일 디스크립터는 자식 프로세스에 상속됩니다. 하나의 파일을 한 프로세스나 여러 프로세스가 여러 번 열면, 각 open은 새 파일 디스크립터를 반환합니다. 같은 파일에 대한 서로 다른 파일 디스크립터들은 각각 별도의 close 호출로 독립적으로 닫히며, 파일 위치를 공유하지 않습니다. 추가 과제를 구현할 때는 0부터 시작하는 정수를 반환하는 Linux 방식을 따르는 것이 좋습니다.

int filesize (int fd);

fd로 열린 파일의 크기를 바이트 단위로 반환합니다.

int read (int fd, void *buffer, unsigned size);

fd로 열린 파일에서 size 바이트를 읽어 buffer에 저장합니다. 실제로 읽은 바이트 수를 반환합니다. 파일 끝에서는 0을 반환하고, 파일 끝이 아닌 다른 이유로 읽을 수 없으면 -1을 반환합니다. fd 0은 input_getc()를 사용해 키보드에서 읽습니다.

int write (int fd, const void *buffer, unsigned size);

buffer에서 열린 파일 fd로 size 바이트를 씁니다. 실제로 쓴 바이트 수를 반환합니다. 일부 바이트를 쓸 수 없으면 이 값은 size보다 작을 수 있습니다. 보통 파일 끝을 넘어 쓰면 파일이 확장되지만, 기본 파일 시스템에는 파일 증가가 구현되어 있지 않습니다. 기대 동작은 파일 끝까지 가능한 만큼 쓰고 실제로 쓴 바이트 수를 반환하거나, 전혀 쓸 수 없으면 0을 반환하는 것입니다. fd 1은 콘솔에 씁니다. 콘솔에 쓰는 코드는 적어도 size가 수백 바이트보다 크지 않은 한 buffer 전체를 한 번의 putbuf() 호출로 써야 합니다. 더 큰 버퍼를 쪼개는 것은 합리적입니다. 그렇지 않으면 서로 다른 프로세스가 출력한 텍스트 줄이 콘솔에서 섞여 사람이 읽기도 어렵고 채점 스크립트도 혼란스러워질 수 있습니다.

void seek (int fd, unsigned position);

열린 파일 fd에서 다음에 읽거나 쓸 바이트 위치를 position으로 바꿉니다. position은 파일 시작부터의 바이트 수입니다. 따라서 위치 0은 파일의 시작입니다. 현재 파일 끝을 넘는 위치로 seek하는 것은 오류가 아닙니다. 이후 읽기는 파일 끝을 뜻하는 0바이트를 얻습니다. 이후 쓰기는 파일을 확장하고, 아직 쓰지 않은 빈 구간은 0으로 채웁니다. 다만 Pintos의 파일은 파일 시스템 확장 과제가 완료되기 전까지 고정 길이이므로, 파일 끝을 넘어 쓰면 오류가 반환됩니다. 이 의미론은 파일 시스템에 구현되어 있으며, 시스템 콜 구현에서 별도의 특별한 작업은 필요하지 않습니다.

unsigned tell (int fd);

열린 파일 fd에서 다음에 읽거나 쓸 바이트의 위치를 파일 시작부터의 바이트 수로 반환합니다.

void close (int fd);

파일 디스크립터 fd를 닫습니다. 프로세스가 exit하거나 종료되면, 열린 모든 파일 디스크립터가 각 디스크립터에 대해 이 함수를 호출한 것처럼 암묵적으로 닫힙니다.

파일에는 다른 시스템 콜도 정의되어 있습니다. 지금은 무시하세요. 그중 일부는 프로젝트 3에서 구현하고, 나머지는 이 로컬 번역 범위 밖의 이후 과제에서 구현하게 됩니다. 따라서 시스템을 설계할 때 확장성을 염두에 두세요.

여러 사용자 프로세스가 동시에 시스템 콜을 호출할 수 있도록 시스템 콜을 동기화해야 합니다. 특히 filesys 디렉터리에 제공된 파일 시스템 코드를 여러 스레드가 동시에 호출하는 것은 안전하지 않습니다. 시스템 콜 구현은 파일 시스템 코드를 critical section으로 취급해야 합니다. process_exec()도 파일에 접근한다는 점을 잊지 마세요. 지금은 filesys 디렉터리의 코드를 수정하지 않는 것을 권장합니다.

lib/user/syscall.c에는 각 시스템 콜에 대한 사용자 수준 함수가 제공되어 있습니다. 이 함수들은 사용자 프로세스가 C 프로그램에서 각 시스템 콜을 호출할 수 있는 방법을 제공합니다. 각 함수는 약간의 inline assembly 코드를 사용해 시스템 콜을 호출하고, 필요한 경우 시스템 콜의 반환값을 반환합니다.

이 부분을 끝낸 뒤부터 Pintos는 매우 견고해야 합니다. 사용자 프로그램이 무엇을 하더라도 OS가 crash되거나, panic하거나, assertion에 실패하거나, 그 밖의 방식으로 오작동해서는 안 됩니다. 이 점은 특히 중요합니다. 테스트는 여러 방식으로 시스템 콜을 깨뜨리려고 시도할 것입니다. 모든 corner case를 생각하고 처리해야 합니다. 사용자 프로그램이 OS를 멈출 수 있는 유일한 방법은 halt 시스템 콜을 호출하는 것뿐이어야 합니다.

시스템 콜에 잘못된 인자가 전달되었을 때 허용되는 처리 방식에는 오류 값 반환(값을 반환하는 호출의 경우), 정의되지 않은 값 반환, 프로세스 종료가 포함됩니다.
