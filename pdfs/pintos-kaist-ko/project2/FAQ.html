FAQ

코드를 얼마나 작성해야 하나요?

아래는 git diff --stat으로 만든 레퍼런스 솔루션의 요약입니다. 마지막 줄은 삽입된 줄 수와 삭제된 줄 수의 합계를 보여 줍니다. 변경된 줄은 삽입 1줄과 삭제 1줄로 모두 계산됩니다.

레퍼런스 솔루션은 가능한 풀이 중 하나일 뿐입니다. 다른 풀이도 많이 가능하며, 그중 상당수는 레퍼런스 솔루션과 크게 다를 수 있습니다. 좋은 풀이가 레퍼런스 솔루션이 수정한 모든 파일을 수정하지 않을 수도 있고, 반대로 레퍼런스 솔루션이 수정하지 않은 파일을 수정할 수도 있습니다. 이 통계에는 추가 요구사항 구현도 포함되어 있습니다. 참고로 약 150줄은 추가 과제와 관련된 코드입니다.

pintos -p file -- -q를 실행하면 커널이 항상 panic합니다.

파일 시스템을 pintos -f로 포맷했나요?

파일 이름이 너무 길지는 않나요? 파일 시스템은 파일 이름을 14자로 제한합니다.

pintos -p ../../examples/echo -- -q 같은 명령은 이 제한을 넘습니다.

대신 pintos -p ../../examples/echo:echo -- -q를 사용해서 파일을 echo라는 이름으로 넣으세요.

파일 시스템이 가득 찼나요?

파일 시스템에 이미 파일이 16개 들어 있나요? 기본 Pintos 파일 시스템은 파일 16개 제한이 있습니다.

파일 시스템이 너무 단편화되어 해당 파일을 넣을 연속 공간이 부족할 수도 있습니다.

pintos -p ../file --를 실행했는데 file이 복사되지 않습니다.

기본적으로 파일은 명령에서 참조한 이름 그대로 기록됩니다. 따라서 이 경우 복사된 파일 이름은 ../file이 됩니다. 아마 원하는 명령은 pintos -p ../file:file --일 것입니다.

모든 사용자 프로그램이 page fault로 죽습니다.

인자 전달을 구현하지 않았거나 올바르게 구현하지 않았을 때 이런 일이 발생합니다. 사용자 프로그램용 기본 C 라이브러리는 스택에서 argv를 읽으려고 합니다. 스택이나 레지스터가 제대로 설정되어 있지 않으면 page fault가 발생합니다.

모든 사용자 프로그램이 system call!을 출력하고 죽습니다.

다른 동작을 보기 전에 시스템 콜을 구현해야 합니다. 정상적인 프로그램은 거의 항상 최소 하나의 시스템 콜, 보통 exit(), 을 호출하고 대부분은 그보다 더 많이 호출합니다. 특히 printf()는 write 시스템 콜을 호출합니다. 기본 시스템 콜 핸들러는 system call!을 출력하고 프로그램을 종료합니다. 그 전까지는 hex_dump()를 사용해 인자 전달이 올바르게 구현되었는지 확인할 수 있습니다.

사용자 프로그램을 어떻게 디스어셈블하나요?

objdump 유틸리티는 사용자 프로그램 전체나 오브젝트 파일을 디스어셈블할 수 있습니다. objdump -d file처럼 실행하세요. 개별 함수를 디스어셈블하려면 GDB의 disassemble 명령을 사용할 수 있습니다.

왜 많은 C include 파일이 Pintos 프로그램에서 동작하지 않나요?

Pintos 프로그램에서 libfoo를 사용할 수 있나요?

제공된 C 라이브러리는 매우 제한적입니다. 실제 운영체제의 C 라이브러리에서 기대하는 많은 기능이 들어 있지 않습니다. C 라이브러리는 I/O와 메모리 할당을 위해 시스템 콜을 수행해야 하므로, 특정 운영체제와 아키텍처에 맞게 만들어져야 합니다. 물론 모든 함수가 시스템 콜을 하는 것은 아니지만, 보통 라이브러리는 하나의 단위로 컴파일됩니다.

사용하려는 라이브러리는 Pintos가 구현하지 않은 C 라이브러리 기능을 사용할 가능성이 큽니다. Pintos에서 동작하게 만들려면 적어도 어느 정도 포팅 작업이 필요할 것입니다. 특히 Pintos 사용자 프로그램 C 라이브러리에는 malloc() 구현이 없습니다.

새 사용자 프로그램은 어떻게 컴파일하나요?

src/examples/Makefile을 수정한 뒤 make를 실행하세요.

디버거에서 사용자 프로그램을 실행할 수 있나요?

가능하지만 몇 가지 제한이 있습니다. GDB를 참고하세요.

tid_t와 pid_t의 차이는 무엇인가요?

tid_t는 커널 스레드를 식별합니다. 그 커널 스레드는 process_fork()로 만들어져 사용자 프로세스를 실행하고 있을 수도 있고, thread_create()로 만들어져 사용자 프로세스를 실행하지 않을 수도 있습니다. tid_t는 커널에서만 사용하는 데이터 타입입니다.

pid_t는 사용자 프로세스를 식별합니다. 사용자 프로세스와 커널이 exec, wait 시스템 콜에서 사용합니다. tid_t와 pid_t에 어떤 타입을 쓸지는 적절히 선택하면 됩니다. 기본적으로 둘 다 int입니다. 둘을 일대일로 매핑해서 같은 값이 같은 프로세스를 가리키게 할 수도 있고, 더 복잡한 매핑을 사용할 수도 있습니다. 설계 선택에 달려 있습니다.

struct file *을 캐스팅해서 파일 디스크립터로 써도 되나요?

struct thread *를 pid_t로 캐스팅해도 되나요?

아니요. pid_t와 file descriptor는 포인터 타입보다 작기 때문에, 포인터를 직접 캐스팅해서 안전하게 담을 수 없습니다.

프로세스당 열 수 있는 파일 수에 최댓값을 둬도 되나요?

임의의 제한을 두지 않는 편이 좋습니다. 필요하다면 프로세스당 열린 파일 128개 제한을 둘 수 있습니다. 하지만 추가 요구사항을 구현하려면 제한이 없어야 합니다.

열려 있는 파일이 삭제되면 어떻게 되나요?

파일에 대해 표준 Unix 의미론을 구현해야 합니다. 즉 어떤 파일이 삭제되더라도, 그 파일에 대한 파일 디스크립터를 가진 모든 프로세스는 그 디스크립터를 계속 사용할 수 있어야 합니다. 따라서 그 파일에서 읽고 쓸 수 있습니다. 파일에는 더 이상 이름이 없고 다른 프로세스가 새로 열 수는 없지만, 그 파일을 참조하는 모든 파일 디스크립터가 닫히거나 머신이 종료될 때까지 파일은 계속 존재합니다.

4 kB보다 큰 스택 공간이 필요한 사용자 프로그램은 어떻게 실행하나요?

각 프로세스에 한 페이지보다 많은 스택 공간을 할당하도록 스택 설정 코드를 수정할 수 있습니다. 다음 프로젝트에서는 더 나은 해결책을 구현하게 됩니다.
