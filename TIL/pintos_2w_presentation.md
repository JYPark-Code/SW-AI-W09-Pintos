## 2 슬라이드 — Git CRLF & 파일 권한 노이즈 폭발

"첫 번째 사례입니다. 코드를 한 줄도 안 고쳤는데 PR diff에 2만 3천 줄이 잡혔습니다.

`git diff --stat` 보시면 insertions와 deletions가 정확히 같은 23,517 — 라인 내용이 아니라 줄 끝 문자가 통째로 교체됐다는 시그니처입니다. 리눅스의 LF가 윈도우의 CRLF로 바뀐 거죠. 여기에 Windows와 Linux 사이 실행 권한 비트 차이까지 겹쳐서 리포 전체가 빨간 Modified로 도배됐고, 코드 리뷰가 불가능한 상태였습니다."


## 3 슬라이드 — 두 가지 봉인 조치

"원인이 두 개니까 조치도 두 개를 동시에 걸었습니다.

첫째, 리포에 `.gitattributes`를 추가해서 텍스트 파일 줄 끝을 LF로 강제했습니다. 둘째, 각 팀원 로컬에 `core.filemode=false`를 적용해서 권한 비트 차이를 git이 무시하게 했습니다.

테스트해 보니 둘 중 하나만 적용하면 노이즈 절반이 그대로 살아남았습니다. 두 개 다 걸어야 완전히 차단됐습니다. 원인이 둘이면 조치도 둘이라는 게 핵심입니다."


## 4 슬라이드 — process_wait 설계 대립: Polling vs Semaphore

"두 번째 주제, `process_wait` 시스템 콜 설계입니다.

자식 종료를 기다리는 방식을 두고 팀 안에서 의견이 갈렸습니다. 코드가 짧은 `timer_sleep` 폴링이냐, 복잡하지만 정확한 semaphore냐였습니다.

폴링은 단순해 보이지만 함정이 있습니다. 자식 종료 시점의 race condition과 use-after-free를 막으려면 결국 또 다른 동기화 메커니즘이 필요해집니다. 단순함이 환상이었던 거죠."


## 5 슬라이드 — 팀의 누적 자산이 곧 설계 근거

"팀이 semaphore를 채택한 결정적 이유는 우리가 이미 잘 다루는 도구였기 때문입니다.

CSAPP Proxy Lab에서 semaphore로 큐를 직접 구현해 본 경험이 있었고, 바로 지난주 Project 1에서 Priority Scheduling을 구현하며 내부 동작을 팀 전원이 이해한 상태였습니다.

새 도구를 억지로 도입할 이유가 없었습니다. 익숙한 semaphore로 추가 학습 비용 없이 안전한 동기화를 만들었고, wait 관련 테스트를 단번에 통과했습니다.

기술적 우위만이 아니라 팀의 누적 경험이 가장 강력한 설계 근거가 된다 — 이게 오늘 발표의 핵심입니다. 이상입니다."
